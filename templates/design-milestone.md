# Design Milestone Template

## Usage
This template is used by /pde:new-project and /pde:migrate when seeding a PDE project roadmap with design phases.

Design phases should be grouped at the start of the milestone (before engineering phases). Each design phase maps to a PDE agent for PDE execution. Design phases use hybrid verification: PDE verifier confirms artifacts exist, plus PDE critique runs as an additional quality gate. Critical critique findings mean the phase is not complete.

## Adaptive Phase Selection

### Software Product
- Phase: Product Brief (/pde:brief)
- Phase: User Flows (/pde:flows)
- Phase: Design System (/pde:system)
- Phase: Wireframes (/pde:wireframe)
- Phase: Design Review (/pde:critique + /pde:hig)
- Phase: Design Iteration (/pde:iterate)
- Phase: Design Handoff (/pde:handoff)

### Hardware Product
- Phase: Product Brief (/pde:brief)
- Phase: User Flows (/pde:flows) [if product has software interface]
- Phase: Hardware Specification (/pde:hardware)
- Phase: Design Review (/pde:critique)
- Phase: Design Handoff (/pde:handoff --hardware)

### Hybrid Product
- Phase: Product Brief (/pde:brief)
- Phase: User Flows (/pde:flows)
- Phase: Design System (/pde:system)
- Phase: Wireframes (/pde:wireframe)
- Phase: Hardware Specification (/pde:hardware)
- Phase: Design Review (/pde:critique + /pde:hig)
- Phase: Design Iteration (/pde:iterate)
- Phase: Design Handoff (/pde:handoff)

### Strategy Add-on (any product type)
- Phase: Competitive Analysis (/pde:competitive) [insert after brief]
- Phase: Opportunity Evaluation (/pde:opportunity) [insert after competitive]

### API-Only Product (no UI)
- Phase: Product Brief (/pde:brief)
- Phase: User Flows (/pde:flows) [API consumer flows]
[No design system, wireframe, or visual phases]

## Phase Ordering Rules
1. Brief is always first
2. Strategy phases (competitive, opportunity) follow brief if included
3. Flows before wireframes (flow informs screen list)
4. Design system before wireframes (tokens inform visual output)
5. Hardware spec can parallel wireframes (independent)
6. Critique/HIG after artifact-producing phases
7. Iterate after critique
8. Handoff is always last design phase

## Integration Notes
- Each design phase maps to a `pde-*.md` agent file in `~/.claude/agents/`
- PDE's execute-phase spawns the corresponding PDE agent via Task()
- Design phases support full auto-advance (discuss -> plan -> execute)
- design-manifest.json is updated after each design phase completes
- Hybrid verification: PDE verifier checks artifact existence + PDE critique as quality gate
- Critical critique findings = phase not complete (blocks advancement)
