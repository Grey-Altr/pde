**Step 4/7: Evaluate wireframes across four perspectives**

Load evaluation references:
- `@references/design-principles.md` -- Nielsen H1-H10, Shneiderman S1-S8, Norman's principles, Gestalt laws, Fitts/Hick/Miller
- `@references/wcag-baseline.md` -- WCAG 2.2 Level A/AA criteria
- `@templates/critique-report.md` -- output format reference

If --focused flag present: evaluate only the named perspective(s). Otherwise evaluate all four.

For each wireframe file in WIREFRAME_FILES:
  Read the wireframe HTML content.
  Determine fidelity from FIDELITY_MAP for severity calibration.

  **Perspective 1: UX / Usability** (weight 1.5x)
  - Frameworks: Nielsen H1-H10, Shneiderman S1-S8
  - Evaluate against: BRIEF_CONTEXT (personas, goals), FLOWS_CONTEXT (journey steps for this screen), wireframe layout
  - Key questions: Does screen serve persona's goal? Feedback for actions (H1)? Consistent interaction model (H4, S1)? Error recovery (H3, H9, S5, S6)? Clear affordances (Norman)?
  - Severity escalation: H1 violation on screens with async operations is never below Major
  - If SEQUENTIAL_THINKING_AVAILABLE: use mcp__sequential-thinking__think for deeper per-finding reasoning

  **Perspective 2: Visual Hierarchy** (weight 1.0x)
  - Frameworks: Gestalt principles, Norman's principles, Fitts's Law, Hick's Law, Miller's Law
  - Evaluate against: wireframe visual layout, design token application, information density
  - Key questions: Primary action visually prominent? Related elements grouped (Gestalt proximity)? Cognitive load appropriate (Miller, Hick)? Interactive elements distinguishable (Gestalt similarity)? Primary actions satisfy Fitts's Law? Whitespace creates groupings (Gestalt common region)?

  **Perspective 3: Accessibility** (weight 1.5x)
  - Frameworks: WCAG 2.2 Level A/AA, POUR principles
  - If AXE_AVAILABLE: run automated WCAG scan on wireframe HTML, tag findings with "[Enhanced by Axe MCP]"
  - If AXE_AVAILABLE is false: use manual checklist from wcag-baseline.md
  - Key questions: Semantic landmarks (role=banner, main, nav, contentinfo)? Accessible labels on interactives? Color contrast sufficient (4.5:1 text, 3:1 large)? Focus order logical? aria-live regions for dynamic updates? Target size >= 24x24 CSS pixels (WCAG 2.5.8)?
  - WCAG 2.2 new criteria to check: 2.4.11 Focus Not Obscured, 2.5.7 Dragging Movements, 2.5.8 Target Size, 3.2.6 Consistent Help, 3.3.7 Redundant Entry, 3.3.8 Accessible Authentication
  - Fidelity calibration: At lofi, color contrast findings are "nit" (placeholder colors only); at hifi they are major/critical. See fidelity-severity table in research.

  **Perspective 4: Business Alignment** (weight 1.0x)
  - Evaluate against: BRIEF_CONTEXT (product type, goals, personas, constraints, scope boundaries), FLOWS_CONTEXT (business outcomes per journey)
  - Key questions: Screens support user journeys from flows? Feature emphasis matches brief positioning? Any out-of-scope features present (check brief Scope Boundaries)? Persona goals align with primary CTAs? Product type constraints visible in interaction model? Brief constraints (offline-first, platform constraints) reflected?

**Fidelity-severity calibration table** (include in workflow for executor reference):
| Finding Type | Lo-fi Severity | Mid-fi Severity | Hi-fi Severity |
| Missing screen for journey step | Critical | Critical | Critical |
| Color contrast failure | Skip | Minor (placeholder) | Major/Critical |
| CTA prominence (visual weight) | Minor (structure only) | Major | Major |
| ARIA label missing | Major (structure visible) | Major | Critical |
| Token not applied | Skip | Minor | Major |

**Score calculation per perspective:**
```
Score = 100 - (sum of finding penalties)
  Critical finding: -25 points
  Major finding: -10 points
  Minor finding: -4 points
  Nit: -1 point
  Floor: 0 (never negative)
```

**Finding format** (every finding MUST have all fields):
- Location: screen.html > section > element (CSS-selector-like path)
- Severity: critical | major | minor | nit
- Effort: quick-fix | moderate | significant
- Issue: specific description of what is wrong
- Suggestion: concrete actionable fix with specific values (e.g., "Change font-size from var(--font-size-xs) to var(--font-size-sm)" NOT "increase font size")
- Reference: the standard/principle supporting this finding (e.g., "Nielsen H1", "WCAG 2.5.8", "Gestalt proximity")
- Perspective: which perspective produced this finding
- Weight: perspective weight (1.0x or 1.5x)

**"What Works" identification** (required by success criterion 4):
For each perspective, identify 1-3 specific intentional design decisions that are correct and should be preserved during iteration. Format:
| Element | What's Working | Perspective | Keep It |
| {screen.element} | {specific observation} | {perspective} | Yes -- do not change in iteration |
