## Summary

Phase 6 covers two directories: `templates/` (55 files across subdirectories) and `references/` (30 files including one subdirectory). A comprehensive grep audit — `grep -rni "gsd\|get-shit-done"` — across both directories returns **zero results**. The templates and references are already GSD-free.

The phase success criteria also requires that a fresh project initialized via `/pde:new-project` produces `.planning/` files with zero GSD strings. The `.planning/` output files (`STATE.md`, `PROJECT.md`, `REQUIREMENTS.md`, `ROADMAP.md`) are generated by workflows from the template files, and those template files contain no GSD strings. The UI components (`lib/ui/splash.cjs`, `lib/ui/components.cjs`) that produce stage banners at runtime already display PDE branding — "Platform Development Engine" in `splash.cjs` and generic stage labels passed as arguments from workflows. All workflow files (the callers) already pass PDE-named stage strings like "QUESTIONING", "RESEARCHING", "PLANNING PHASE {X}" — no GSD prefix appears in any banner call.

The primary work for this phase is verification, not remediation. Both plans (06-01 and 06-02) should take the form of audits that produce evidence confirming the zero-GSD state, followed by a fresh `/pde:new-project` smoke test to confirm the end-to-end artifact chain is GSD-free.

**Primary recommendation:** Structure both plans as audit-and-confirm tasks. If the audit finds any GSD remnants (unlikely given pre-phase grep), fix in place. The main deliverable is a verified, documented statement that templates and references are PDE-branded.

## Architecture Patterns

### Recommended Audit Structure

```
For each target directory (templates/, references/):
1. grep -rni "gsd\|get-shit-done" <dir>   → expect zero output
2. grep -rni "/gsd:" <dir>                 → expect zero output
3. grep -rni "~/.gsd" <dir>               → expect zero output
4. grep -rni "gsd-tools" <dir>            → expect zero output
5. grep -rni "get.shit.done" <dir>        → expect zero output
```

### Fresh Project Smoke Test Pattern

The success criterion "a fresh project initialized via /pde:new-project produces .planning/ files with zero GSD strings" should be verified by:

1. Read key generated template files (state.md, project.md, roadmap.md, requirements.md, summary.md)
2. Confirm none contain GSD strings in their template bodies
3. Since generation is template-driven, clean templates guarantee clean output

The actual end-to-end smoke test (running `/pde:new-project` in a real session) is a human verification step — it cannot be automated within the plan executor. The plan should call this out explicitly as a human check.

### Anti-Patterns to Avoid

- **Blind string replacement:** Do not run `sed -i 's/gsd/pde/g'` across all files. Some template files intentionally use generic placeholder text (e.g., `[Project Name]`) and any GSD that appears in content examples (e.g., in the `requirements.md` template's example section showing an older version number) must be evaluated in context.
- **Missing JSON files:** The `templates/config.json` and `templates/design-manifest.json` and `templates/handoff-manifest.json` and `templates/sources-manifest.json` are JSON, not markdown. The grep must cover all file types, not just `.md`.

## Common Pitfalls

### Pitfall 1: Partial File-Type Coverage
**What goes wrong:** Grep only covers `.md` files, misses `.json` templates.
**Why it happens:** Phase description mentions "template files" which sounds like markdown-only.
**How to avoid:** Run grep without `--include` file-type filter, or explicitly add `--include="*.json"`.
**Warning signs:** Audit reports zero hits but JSON files were not in scope.

### Pitfall 2: Treating Research Files as In-Scope
**What goes wrong:** Executor finds GSD strings in `.planning/research/STACK.md` and reports a violation.
**Why it happens:** The research docs in `.planning/research/` were written during initial project setup and intentionally discuss "GSD" as the source system. These are historical research artifacts, not templates or references.
**How to avoid:** Scope the audit explicitly to `templates/` and `references/` directories only. Phase 7 handles the global codebase grep.
**Warning signs:** Audit report includes `.planning/` paths.

### Pitfall 3: Assuming "Already Done" Without Evidence
**What goes wrong:** Plan executor skips the audit because "prior phases probably handled it."
**Why it happens:** The templates were apparently already copied in Phase 3 (which copied "lib/ui, references, and templates from PDE reference"). Those copies appear clean.
**How to avoid:** Run the audit explicitly and record the evidence. Zero-hit grep output IS the deliverable. Document it.

### Pitfall 4: Banner Stage Names Are Passed at Call Sites
**What goes wrong:** Executor searches templates for banner content like "QUESTIONING" or "PLANNING PHASE" and flags them as needing PDE branding.
**Why it happens:** Misunderstanding of how banners work. The `render.cjs banner` component takes a free-form string argument; stage names are hardcoded in workflow `.md` files as the argument passed to the binary, not in templates.
**How to avoid:** Understand the architecture: templates produce `.planning/` markdown artifacts; banners are produced by workflows calling `render.cjs banner "STAGE NAME"`. Workflows are Phase 3 scope (already complete). Templates scope is the `.planning/` artifact content.
