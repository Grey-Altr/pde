### INT-02 (High) — Cross-Domain Dependency Map column mismatch

**Defect location:** `workflows/brief.md`, Step 7/7, "Update root DESIGN-STATE and manifest" section

**Root cause:** The workflow instructs Step 7 to write this row to the Cross-Domain Dependency Map:
```
| BRF | Design Brief | strategy | v{N} | -- | {date} |
```
That row has **6 columns**. The actual table schema in `DESIGN-STATE.md` has **4 columns**:
```markdown
| Artifact | Domain | Depends On | Status |
|----------|--------|------------|--------|
```
The workflow line is copied in from an earlier draft schema that included `Name`, `Domain`, `File`, `Version`, `Depends On`, `Updated` columns. The live template (`templates/design-state-root.md`) and the initialized `.planning/design/DESIGN-STATE.md` both use the 4-column schema.

**Fix:** Change the BRF row instruction in Step 7 of `workflows/brief.md` to:
```
| BRF | strategy | -- | current |
```
Column mapping: Artifact=BRF, Domain=strategy, Depends On=--, Status=current.

**Impact:** Without this fix, the first real `/pde:brief` run inserts a 6-column data row into a 4-column table, producing broken Markdown that will corrupt `DESIGN-STATE.md` for all downstream phases 14-20.

**Confidence:** HIGH — confirmed by reading both files.

### INT-01 (Medium) — lock-status subcommand not implemented

**Defect location:** `bin/lib/design.cjs` and `bin/pde-tools.cjs`

**Root cause:** The `pde-tools.cjs` header docs (lines 55-62) list `design lock-status` as an available subcommand. The ROADMAP Phase 13.1 success criteria also name it. However:
- `design.cjs` exports: `cmdEnsureDirs`, `cmdManifestRead`, `cmdManifestUpdate`, `cmdArtifactPath`, `cmdTokensToCss`, `cmdCoverageCheck`, `cmdLockAcquire`, `cmdLockRelease` — no `cmdLockStatus`
- The `case 'design':` block in `pde-tools.cjs` (lines 494-516) routes 8 subcommands but has no `lock-status` branch

**What lock-status should do:** Return a JSON object describing the current write lock state: whether a lock is held, the owner, when it expires, and whether it is stale. This is a read-only operation that does NOT modify DESIGN-STATE.md.

**Implementation pattern (mirrors existing lock functions):**
```javascript
// In design.cjs — add after cmdLockRelease

function cmdLockStatus(cwd, raw) {
  const stateFilePath = path.join(cwd, '.planning', 'design', 'DESIGN-STATE.md');
  if (!fs.existsSync(stateFilePath)) {
    output({ locked: false, owner: null, since: null, expires: null, stale: false }, raw);
    return;
  }
  const content = fs.readFileSync(stateFilePath, 'utf-8');
  const writeLockSectionPattern = /(### Write Lock\n\| Locked By \| Since \| Expires \|\n\|[-| ]+\|\n)([\s\S]*?)(?=\n##|\n### (?!Write Lock)|$)/;
  const match = content.match(writeLockSectionPattern);
  const tableBody = match ? match[2] : '';
  const dataRowPattern = /^\| (?!<!--)([^|]+) \| ([^|]+) \| ([^|]+) \|$/m;
  const dataRowMatch = tableBody.match(dataRowPattern);
  if (!dataRowMatch) {
    output({ locked: false, owner: null, since: null, expires: null, stale: false }, raw);
    return;
  }
  const owner = dataRowMatch[1].trim();
  const since = dataRowMatch[2].trim();
  const expires = dataRowMatch[3].trim();
  const stale = new Date(expires) <= new Date();
  output({ locked: !stale, owner, since, expires, stale }, raw);
}
```

**pde-tools.cjs routing addition** (inside `case 'design':` block):
```javascript
} else if (subcommand === 'lock-status') {
  design.cmdLockStatus(cwd, raw);
}
```

**Export addition** (in `module.exports`):
```javascript
cmdLockStatus,
```

**Self-test addition** — add one check to Test group 2:
```javascript
check('cmdLockStatus returns locked: false when no lock held', () => {
  // After releaseWriteLock, no lock row exists
  const result = JSON.parse(/* capture output */);  // call via cmdLockStatus directly
  assert.strictEqual(result.locked, false);
});
```

Note: The self-test uses inline `check()` calls, not external test files. The new lock-status assertion should be placed after the existing lock release test.

**Confidence:** HIGH — fully derived from reading the existing `acquireWriteLock` / `releaseWriteLock` parsing patterns in `design.cjs`.

## Architecture Patterns

### Pattern: Extend existing cmd* wrappers in design.cjs

`design.cjs` follows the established `cmd*` wrapper pattern from `state.cjs`:
1. Implement the core logic function (e.g., `lockStatus(cwd)`)
2. Add a `cmdLockStatus(cwd, raw)` wrapper that calls `output(result, raw)`
3. Export both from `module.exports`
4. Add routing in `pde-tools.cjs` `case 'design':` block
5. Add assertion in `runSelfTest()`

This is the only pattern for new design subcommands. Do not break it.

### Pattern: Single-wave hotfix plan

All five fixes are independent — none depends on another. They can be sequenced as:
1. INT-02 (workflows/brief.md) — highest severity, simplest fix
2. INT-01 (design.cjs + pde-tools.cjs) — new function, most code
3. INT-03 (design.cjs DOMAIN_DIRS + self-test label) — one-liner + label
4. Metadata fix (13-01-SUMMARY.md frontmatter) — doc-only
5. Nyquist fix (12-VALIDATION.md frontmatter + checklist) — doc-only

All five can be committed in a single wave. Run `node bin/lib/design.cjs --self-test` after tasks 2 and 3.

### Anti-Patterns to Avoid
- **Do not change the DESIGN-STATE.md schema** to match the old 6-column row. The schema IS 4 columns. The workflow row is wrong.
- **Do not create a separate `.lock` file** for lock-status. The lock state lives in the DESIGN-STATE.md Write Lock table by design decision (Phase 12).
- **Do not touch 13-VERIFICATION.md** — BRF-01 and BRF-02 are verified as SATISFIED there. The metadata fix only touches the SUMMARY frontmatter.

## Common Pitfalls

### Pitfall 1: Fixing the template instead of the workflow

**What goes wrong:** The INT-02 fix might be applied to `templates/design-state-root.md` (adding columns to match the 6-column row) instead of fixing the row in `workflows/brief.md`.

**Why it happens:** Both files are relevant to the Cross-Domain Dependency Map. It's tempting to "upgrade" the template schema.

**How to avoid:** The 4-column schema is the intended schema. The examples in the template comments also use 4 columns. The workflow row is the bug, not the template.

**Warning signs:** If you find yourself adding `Name` or `Version` columns to the DESIGN-STATE table header — stop. You're going the wrong direction.

### Pitfall 2: Adding hardware to DOMAIN_DIRS without updating the self-test label

**What goes wrong:** The self-test label says "creates all 6 domain subdirectories" — after adding hardware it checks 7. The test still passes (it loops `DOMAIN_DIRS`) but the label is wrong.

**How to avoid:** Update the label to "creates all domain subdirectories" in the same commit as the DOMAIN_DIRS change.

### Pitfall 3: Implementing lock-status as a write operation

**What goes wrong:** Accidentally inserting a lock row as a side effect of reading status.

**How to avoid:** `cmdLockStatus` must be read-only. It calls `fs.readFileSync` but never `fs.writeFileSync`. Mirror the read-only portion of `acquireWriteLock` only.

### Pitfall 4: Using the wrong frontmatter set syntax

**What goes wrong:** `pde-tools.cjs frontmatter set` might not handle YAML array syntax correctly for `[BRF-01, BRF-02]`.

**How to avoid:** Use the Write tool directly to edit `13-01-SUMMARY.md` frontmatter. Insert the `requirements-completed` field manually in the correct YAML position. Verify with `node bin/pde-tools.cjs frontmatter get .planning/phases/13-problem-framing-pde-brief/13-01-SUMMARY.md --field requirements-completed` after the edit.
