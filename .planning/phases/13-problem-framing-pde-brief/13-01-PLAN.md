**File structure:**

1. **`<purpose>` block** — One paragraph: "Generate a structured product design brief from PROJECT.md context. Produces BRF-brief-v1.md in .planning/design/strategy/ covering problem statement, product type detection, personas, jobs-to-be-done, goals, constraints, and non-goals.    ```


3. **`<flags>` block** — Document supported flags:
   - `--dry-run` — Show planned output without executing
   - `--quick` — Skip MCP enhancements for faster execution
   - `--verbose` — Show detailed progress and MCP probe results
   - `--no-mcp` — Skip all MCP probes
   - `--force` — Skip confirmation prompt when brief already exists (skill-specific flag)

4. **`<process>` block** — The 7-step pipeline. Each step MUST use the exact `Step N/7:` progress format from skill-style-guide.md.

**Step 1: Initialize design directories**
```bash
INIT=$(node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design ensure-dirs)
if [[ "$INIT" == @file:* ]]; then INIT=$(cat "${INIT#@file:}"); fi
```
Parse result. If error, display error per skill-style-guide.md format and halt.

**Step 2: Check prerequisites**
- Read `.planning/PROJECT.md` using the Read tool
- If PROJECT.md does not exist: ERROR (hard stop, not warning):
  ```
  Error: PROJECT.md not found at .planning/PROJECT.md
    /pde:brief requires a project description to generate a design brief.
    Run /pde:new-project first to initialize your project, then re-run /pde:brief.
  ```
- Optionally read `.planning/REQUIREMENTS.md` if it exists (soft dependency — enriches brief but not required)
- Check if `strategy/BRF-brief-v1.md` already exists:
  - If exists AND `--force` flag NOT present: prompt user "A brief already exists (v1). Generate a new version? (This will create BRF-brief-v2.md)"
  - If exists AND `--force` flag present: auto-increment version
  - If does not exist: proceed with v1
- Determine version number N (1 if first run, max existing + 1 if re-run)

**Step 3: Probe MCP (Sequential Thinking)**
- Check `--no-mcp` and `--no-sequential-thinking` flags first
- If not disabled: probe `mcp__sequential-thinking__think` with test prompt "Analyze the following: test"
- 30-second timeout, 1 retry on failure
- Set `SEQUENTIAL_THINKING_AVAILABLE` boolean
- If `--quick` flag: skip MCP probes entirely

**Step 4: Detect product type (BRF-02)**
- Scan PROJECT.md content for software signals: Node.js, React, Vue, TypeScript, Python, Swift, Kotlin, SQL, API, SaaS, web app, mobile app, CLI, deploy, server, cloud, containerize, npm, pip, app store, web browser, mobile OS, desktop OS
- Scan for hardware signals: PCB, circuit board, 3D print, CAD, BOM, enclosure, firmware, microcontroller, sensor, actuator, embedded, assembly, injection mold, CNC, FDM, tolerance, product box, physical unit, supply chain
- Classification logic:
  - Both software AND hardware signals present → `hybrid`
  - Only hardware signals → `hardware`
  - Only software signals OR ambiguous → `software` (default)
- Resolve to ONE canonical lowercase string: `software`, `hardware`, or `hybrid`
- If Sequential Thinking MCP available: use it for deeper analysis of ambiguous cases
- Detect platform: `web`, `mobile`, `desktop`, `embedded`, `multi-platform` — based on deployment/distribution signals in PROJECT.md
- Record the signals found and the rationale for the classification

**Step 5: Synthesize brief content**
- Using PROJECT.md content (and REQUIREMENTS.md if available), fill every section from `templates/design-brief.md`:
  - **Frontmatter**: Generated date (ISO 8601), Skill: `/pde:brief (BRF)`, Version: `v{N}`, Status: `draft`, Enhanced By: list of MCPs used or `none`
  - **Problem Statement**: 1-2 paragraphs synthesized from PROJECT.md core value, description, and pain points
  - **Product Type**: The canonical type from Step 4, platform, rationale with signals found, and design constraints table:
    - For `software`: Screen resolution targets, browser/OS compatibility, accessibility requirements (WCAG 2.2 AA), touch vs. mouse interaction model, responsive breakpoints
    - For `hardware`: Physical dimensions and tolerances, material constraints, manufacturing process limits, regulatory compliance (CE, UL, FCC), IP rating if applicable
    - For `hybrid`: Combined software + hardware constraints + integration protocol (BLE, USB, API bridge)
  - **Target Users**: Primary and secondary personas with Role, Goal, Pain points, Context — synthesized from PROJECT.md
  - **Jobs to Be Done**: Section placed between Target Users and Constraints — list of JTBD per persona in "When [situation], I want to [motivation], so I can [expected outcome]" format
  - **Constraints**: Table with Constraint, Type (technical/business/regulatory/time), Impact
  - **Success Criteria**: Table with Criterion, Metric, Target
  - **Competitive Context**: Table with Competitor, Strength, Weakness, Differentiation Opportunity
  - **Key Assumptions**: Numbered list with assumption and risk-if-wrong
  - **Scope Boundaries**: In scope list + Out of scope (v1) list with why deferred — this section covers the "non-goals" requirement
  - **Footer**: `*Generated by PDE-OS /pde:brief | {date}*`
- If Sequential Thinking available: use it for persona synthesis and competitive analysis
- Write the complete brief to `.planning/design/strategy/BRF-brief-v{N}.md` using the Write tool

**Step 6: Update domain DESIGN-STATE**
- Check if `strategy/DESIGN-STATE.md` exists in `.planning/design/strategy/`
- If not: create it from `templates/design-state-domain.md`, replacing `{domain_name}` with `strategy` and `{Domain}` with `Strategy`, and `{date}` with current ISO date
- Add BRF artifact row to Artifact Index table:
  ```
  | BRF | Design Brief | /pde:brief | draft | v{N} | {MCP list or "none"} | -- | {date} |
  ```
- If re-run (v2+): update the existing BRF row's Version and Updated columns

**Step 7: Update root DESIGN-STATE and manifest**
- Acquire write lock:
  ```bash
  LOCK=$(node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design lock-acquire pde-brief)
  ```
  Parse `{"acquired": true/false}`. If false, wait 2 seconds and retry (stale locks auto-clear).
- Update root `.planning/design/DESIGN-STATE.md`:
  - Add BRF row to Cross-Domain Dependency Map (if not already present)
  - Update Quick Reference section: `| Product Type | {type} |` and `| Platform | {platform} |`
  - Append to Decision Log: `BRF -- brief complete, product type detected as {type}`
  - Append to Iteration History: `BRF-brief-v{N} | v{N} | Created by /pde:brief | {date}`
- Release write lock (ALWAYS, even on error):
  ```bash
  node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design lock-release
  ```
- Update manifest:
  ```bash
  node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design manifest-update BRF code BRF
  node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design manifest-update BRF name "Design Brief"
  node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design manifest-update BRF type brief
  node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design manifest-update BRF domain strategy
  node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design manifest-update BRF path ".planning/design/strategy/BRF-brief-v{N}.md"
  node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design manifest-update BRF status draft
  node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design manifest-update BRF version {N}
  ```
