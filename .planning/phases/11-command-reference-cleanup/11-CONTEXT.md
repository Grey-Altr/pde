**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Remove or stub dangling command references that break user expectations. Every /pde: command referenced in user-facing output must either exist as a command stub or registered skill, or be clearly marked as planned. No new capabilities — reference hygiene only.

</domain>

<decisions>
## Implementation Decisions

### /pde:recommend handling
- Create a command stub at `commands/recommend.md` that routes to the existing `pde:recommend` skill
- Stub pattern: brief one-liner description of what the command does, then invoke the skill
- `templates/recommendations.md` footer stays as-is: `*Generated by /pde:recommend v1.0*`
- `workflows/new-project.md` success criterion (line 1094) stays as-is — still valid once command stub exists

### v2 design-pipeline command stubs
- Create command stubs for ALL /pde: commands referenced in docs that lack `commands/` entries
- Full list: every /pde: command found in references/ and templates/ that's invokable as a skill but missing from commands/
- Includes: wireframe, system, critique, hig, mockup, handoff, flows, competitive, opportunity, hardware, iterate, brief, setup, test, source, migrate, build, ideate, recommend
- Same stub pattern as /pde:recommend: brief intro describing the command, then route to the corresponding pde:* skill
- Consistent behavior across all command stubs

### Verification approach
- Automated grep audit: extract all /pde: references from workflows/ and references/, cross-reference against commands/ directory + known skills
- Flag any /pde: command that doesn't resolve to either a command stub or a registered skill
- Scope: workflows/ and references/ (user-facing output). Templates included for completeness but secondary
- .planning/ files EXCLUDED from audit scope — immutable historical records (consistent with Phase 7 decision)
- Orphaned references (commands that don't exist and aren't skills): mark as "planned — not yet available" rather than removing

### Claude's Discretion
- Exact one-liner descriptions for each command stub
- Order of stub creation
- Exact grep patterns for the automated audit
- Whether to also audit templates/ or focus strictly on workflows/ + references/

</decisions>
