## Summary

Phase 11 is a reference hygiene phase, not a capability phase. Every `/pde:` command name that appears in `workflows/`, `references/`, or `templates/` must resolve to an entry in `commands/` — either a stub that routes to a workflow, or a stub that describes a skill. This phase creates those stubs for the 20 commands that are referenced but currently missing.

The 20 missing commands fall into two clear groups. Five are core workflow commands (`resume-work`, `debug`, `discuss-milestone`, `join-discord`, `reapply-patches`) that are described as operational in `workflows/help.md` and referenced in other workflows, but lack `commands/*.md` entries. Fifteen are design-pipeline skills (`wireframe`, `system`, `critique`, `hig`, `mockup`, `handoff`, `flows`, `competitive`, `opportunity`, `hardware`, `iterate`, `brief`, `setup`, `test`, `recommend`) that are v2 features documented extensively in `references/` and `templates/` but have no command stubs.

The stub format is fully established by the 34 existing stubs in `commands/`. For core workflow commands, the stub routes to a `workflows/*.md` file via `@${CLAUDE_PLUGIN_ROOT}/workflows/`. For design-pipeline commands that have no backing workflow, the stub describes the skill inline and marks it as v2/future available. The phase also needs to resolve the `resume-work` vs `resume-project` name collision: `commands/resume-project.md` exists but workflows reference `/pde:resume-work`.

**Primary recommendation:** Create 20 command stubs in `commands/` following the established pattern, using `"planned — not yet available (v2)"` in the `<process>` block for design-pipeline commands without workflows, and then run the grep audit to confirm zero dangling references remain.

### Pattern 3: Audit Script Pattern

The audit grep must extract all `/pde:` command names, normalize them, and diff against the `commands/` directory.

```bash
# Step 1: Extract all /pde: command names from workflows/ and references/
grep -roh "/pde:[a-z-]*" workflows/ references/ | sort -u > /tmp/referenced.txt

# Step 2: Extract registered command names from commands/ directory
ls commands/*.md | sed 's|commands/||; s|\.md$||' | sort > /tmp/registered.txt

# Step 3: Show commands referenced but not registered
comm -23 /tmp/referenced.txt /tmp/registered.txt

# A clean audit produces zero output from Step 3.
```

Note: macOS `grep` does not support `-P` (Perl regex). Use `-o -h` with basic regex, or use `ripgrep` (`rg`) which is available in the environment.

### Anti-Patterns to Avoid

- **Removing references instead of adding stubs:** CONTEXT.md explicitly requires stubs or "planned" markers, not deletions. User-facing docs that reference v2 commands are intentional.
- **Creating stubs that point to nonexistent workflows:** v2 stubs must have an inline `<process>` block, not a dangling `@${CLAUDE_PLUGIN_ROOT}/workflows/wireframe.md` reference — that file does not exist.
- **Using `-P` flag with macOS grep:** macOS grep does not support Perl regex. Use `-E` or basic patterns, or use `rg` instead.
- **Auditing .planning/ files:** Explicitly excluded from scope per CONTEXT.md and Phase 7 decision.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cross-reference audit | A custom Node.js script | Simple shell `grep + comm` or `rg` pattern | The audit is a one-time verification step; a 3-line shell sequence is sufficient |
| Command registration | A manifest/registry file | Drop a `.md` file in `commands/` | Claude Code auto-discovers all `.md` files in `commands/`; no separate registration needed |
| v2 feature stubs | A separate "stub registry" | Inline `<process>` block in each stub | One file per command is the established pattern; no separate tracking needed |

**Key insight:** Claude Code discovers commands by scanning the `commands/` directory for `.md` files. The frontmatter `name: pde:{name}` field registers the slash command in the palette. There is no manifest to update.

## Common Pitfalls

### Pitfall 1: resume-work vs resume-project Name Confusion
**What goes wrong:** Creating `commands/resume-work.md` that points to a nonexistent `workflows/resume-work.md`, when the actual workflow is `workflows/resume-project.md`.
**Why it happens:** The command name used in user-facing workflow prose (`/pde:resume-work`) differs from the file name of the backing workflow (`resume-project.md`).
**How to avoid:** `commands/resume-work.md` should point to `@${CLAUDE_PLUGIN_ROOT}/workflows/resume-project.md`. Alternatively, also check if a `commands/resume-project.md` already exists (it does) and determine whether `resume-work` should alias it or replace it.
**Warning signs:** After creating the stub, running the audit still shows `/pde:resume-work` missing if the workflow path in the stub is wrong.

### Pitfall 2: Assuming workflows exist for all core commands
**What goes wrong:** Creating a stub that references `workflows/debug.md` or `workflows/discuss-milestone.md` when those files may not exist.
**Why it happens:** The referenced command is described in `help.md` as operational, but the backing workflow file was never created.
**How to avoid:** Before writing a workflow-backed stub, verify the workflow file exists: `ls workflows/{name}.md`. If missing, use a v2 stub pattern or create a minimal inline workflow.
**Warning signs:** `@${CLAUDE_PLUGIN_ROOT}/workflows/debug.md` resolves to a 404 at runtime.

### Pitfall 3: Audit scope creep into .planning/
**What goes wrong:** The grep audit picks up `/pde:` references inside `.planning/` files (phase documents, STATE.md, CONTEXT.md, etc.) and reports false positives.
**Why it happens:** `.planning/` files freely reference command names as text (e.g., in decision logs).
**How to avoid:** Scope all audit grep commands to `workflows/` and `references/` directories only. Exclude `.planning/` explicitly.
**Warning signs:** Audit reports dozens of "missing" commands that are all in `.planning/` files.

### Pitfall 4: Stubs with wrong frontmatter `name` field
**What goes wrong:** A stub file named `commands/wireframe.md` has `name: pde:wire-frame` (wrong) instead of `name: pde:wireframe` (correct).
**Why it happens:** Typo during mass creation of 20 stubs.
**How to avoid:** Verify frontmatter `name` exactly matches the command as referenced in docs. Run a post-creation grep: `grep "^name:" commands/*.md | sort`.

### Pitfall 5: Omitting `Task` from allowed-tools
**What goes wrong:** New stubs don't include `Task` in `allowed-tools`, breaking commands that need to spawn subagents.
**Why it happens:** Copying an older stub that predates the Phase 3 decision to add `Task` universally.
**How to avoid:** All 34 existing stubs include `Task`. New stubs must include it too. Verify with: `grep -L "Task" commands/*.md`.
