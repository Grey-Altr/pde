**Create workflows/brief-from-github.md:**

**Purpose section:**
```
Fetch a GitHub issue by URL or number, extract its title, body, and labels, and use them to pre-populate the /pde:brief pipeline. The GitHub issue data provides context for the brief generation — it does not replace the full pipeline. Pre-filled sections are marked with [from GitHub #<number>] suffix. Implements GH-03.
```

**Step 0 — Parse --from-github argument:**
The argument can be:
- A full URL: `https://github.com/<owner>/<repo>/issues/<number>` — parse owner, repo, issue_number from URL segments
- A bare number: `42` — use owner/repo from mcp-connections.json
- An `owner/repo#number` format: parse accordingly

Validate that issue_number is a positive integer. Convert string to integer via `parseInt(issueNumber, 10)` (CRITICAL: issue_read requires integer, not string — see research Pitfall 6).

If argument is missing or unparseable, display:
```
Usage: /pde:brief --from-github <issue-url-or-number>

Examples:
  /pde:brief --from-github https://github.com/acme/project/issues/42
  /pde:brief --from-github 42
  /pde:brief --from-github acme/project#42
```
Stop here.

**Step 1 — Load repo info and check GitHub availability:**
```bash
node --input-type=module <<'EOF'
import { createRequire } from 'module';
const req = createRequire(import.meta.url);
const b = req(`${process.env.CLAUDE_PLUGIN_ROOT}/bin/lib/mcp-bridge.cjs`);
const conn = b.loadConnections();
const gh = conn.connections && conn.connections.github;
const repo = gh && gh.repo || '';
const status = gh && gh.status || 'not_configured';
const lookup = b.call('github:get-issue', {});
process.stdout.write(JSON.stringify({ repo, status, toolName: lookup.toolName }) + '\n');
EOF
```

If no owner/repo available (neither from URL parsing nor from mcp-connections.json), display degraded message:
```
GitHub is not connected or no repo configured.
Run /pde:connect github to set up.
Running /pde:brief without GitHub context.
```
Then delegate to the standard brief workflow (@workflows/brief.md) without pre-population. Stop the from-github workflow.

**Step 2 — Fetch issue via MCP:**
Instruct Claude to call the MCP tool returned by bridge.call('github:get-issue') — i.e., `mcp__github__issue_read` — with:
- `method`: `"get"`
- `owner`: parsed owner (from URL or mcp-connections.json)
- `repo`: parsed repo (from URL or mcp-connections.json)
- `issue_number`: parsed integer

The actual MCP call happens in Claude Code's execution context.

If the call fails (issue not found, permissions error, server unavailable), display:
```
Could not fetch GitHub issue #<number>. Error: <error message>
Running /pde:brief without GitHub context.
```
Then delegate to standard brief workflow without pre-population.

**Step 3 — Extract and format issue data for brief pre-population:**
From the issue response, extract:
- `title` → use as seed for `## Problem Statement` section: `<title> [from GitHub #<number>]`
- `body` → use as seed for `## Context` section: format as blockquote with `> ` prefix per line, followed by `[from GitHub #<number>]`
- `labels[]` → use as seeds for `## Constraints` section: each label on its own line as `- <label.name>`, followed by `[from GitHub #<number>]`
- `assignees[]` → include as a note in brief context: `Assignees: <login1>, <login2>`

Display summary:
```
GitHub Issue #<number> loaded:
  Title: <title>
  Labels: <label1>, <label2> (or "none")
  Body: <first 80 chars>...

Pre-populating brief with issue context.
```

**Step 4 — Delegate to main brief pipeline with pre-populated context:**
Set the pre-populated fields as environment context for the brief workflow. The brief workflow should receive this data and use it as starting context for its generation steps. The workflow instruction should tell the executor: "Now run the standard /pde:brief workflow, but use the following pre-populated context from GitHub issue #<number> as starting input for the relevant sections."

Pass through any other flags from $ARGUMENTS (--quick, --dry-run, --verbose, etc.) to the brief workflow.

**Important conventions:**
- All bash blocks use `node --input-type=module` with `createRequire()` pattern
- Never call raw MCP tool names — always look up via bridge.call()
- issue_number MUST be integer, not string
- Pre-populated sections MUST include `[from GitHub #<number>]` marker

**Create workflows/handoff-create-prs.md:**

**Purpose section:**
```
Create GitHub PRs from handoff artifacts. Reads the most recent HND-handoff-spec-v{N}.md from .planning/design/implementation/, constructs PR creation payloads, presents a confirmation list to the user, and only proceeds after explicit user approval. Implements GH-02. Satisfies VAL-03 (write-back confirmation gate).
```

**Step 0 — Initialize:**
Load repo info from mcp-connections.json:
```bash
node --input-type=module <<'EOF'
import { createRequire } from 'module';
const req = createRequire(import.meta.url);
const b = req(`${process.env.CLAUDE_PLUGIN_ROOT}/bin/lib/mcp-bridge.cjs`);
const conn = b.loadConnections();
const gh = conn.connections && conn.connections.github;
const repo = gh && gh.repo || '';
const status = gh && gh.status || 'not_configured';
const lookup = b.call('github:create-pr', {});
process.stdout.write(JSON.stringify({ repo, status, toolName: lookup.toolName }) + '\n');
EOF
```

If repo is empty or status is not `connected`, display:
```
GitHub is not connected or no repo configured.
Run /pde:connect github to set up.
PR creation skipped.
```
Stop here.

**Step 1 — Find handoff artifacts:**
Search for handoff spec files:
```
.planning/design/implementation/HND-handoff-spec-v*.md
```

Use Glob to find all matching files. If none found, display:
```
No handoff specs found. Run /pde:handoff first to generate a handoff spec.
```
Stop here.

Select the latest version (highest v{N} number).

Read the handoff spec to extract:
- Title: from the first `# ` heading or the filename
- Summary: from the first paragraph or `## Summary` section (first 500 chars for PR body)

**Step 2 — Gather PR details:**
Ask the user for the branch name:
```
Which branch contains this work? (the branch must already be pushed to GitHub)
Branch name:
```

Wait for user response. Store as `HEAD_BRANCH`.

Ask for the target branch (with default):
```
Target branch [main]:
```

If user provides a value, use it. Otherwise default to `main`. Store as `BASE_BRANCH`.

Split the `repo` value on `/` to get `owner` and `repoName`.

Construct the PR payload:
- `owner`: parsed owner
- `repo`: parsed repoName
- `title`: from handoff spec title
- `head`: HEAD_BRANCH
- `base`: BASE_BRANCH
- `body`: from handoff spec summary
- `draft`: false
- `maintainer_can_modify`: true

**Step 3 — Display confirmation gate (CRITICAL — no write without user approval):**

Display the full PR details for user review:
```
GitHub PRs to create:

  1. [<title>] <HEAD_BRANCH> -> <BASE_BRANCH> in <owner>/<repoName>
     Body: <first 100 chars of body>...

Create this PR? (y/n)
```

Wait for user response.

If user responds with anything other than `y` or `yes` (case-insensitive):
```
No PRs created.
```
Stop here. Do NOT create any PR.

**Step 4 — Create PR via MCP (only after user confirms):**
Instruct Claude to call the MCP tool returned by bridge.call('github:create-pr') — i.e., `mcp__github__create_pull_request` — with the constructed payload:
- `owner`: owner
- `repo`: repoName
- `title`: PR title
- `head`: HEAD_BRANCH
- `base`: BASE_BRANCH
- `body`: PR body
- `draft`: false
- `maintainer_can_modify`: true

The actual MCP call happens in Claude Code's execution context.

If the call succeeds, display:
```
PR created successfully!
  Title: <title>
  URL: <PR URL from response>
  Branch: <HEAD_BRANCH> -> <BASE_BRANCH>
```

If the call fails, display the error and suggest common fixes:
```
PR creation failed: <error message>

Common causes:
  - Branch '<HEAD_BRANCH>' not pushed to GitHub (run: git push -u origin <HEAD_BRANCH>)
  - Branch '<HEAD_BRANCH>' does not exist
  - A PR already exists for this branch
```

**Important conventions:**
- All bash blocks use `node --input-type=module` with `createRequire()` pattern
- NEVER create a PR without the user explicitly confirming (y/yes)
- Never call raw MCP tool names — always look up via bridge.call()
- Never hardcode owner/repo — read from mcp-connections.json
