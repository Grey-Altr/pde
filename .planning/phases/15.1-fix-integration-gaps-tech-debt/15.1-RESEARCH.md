### INT-01 + FLW-BREAK-01: Dead `hasBrief` instruction in system.md Step 7

**Verified location:** `workflows/system.md` lines 1326 and 1330.

Line 1326 (prose instruction):
> "Merge `hasDesignSystem: true` into the existing coverage object, preserving all other flags (e.g., if `hasBrief` was already `true`, it must remain `true`)."

Line 1330 (bash command template):
```bash
node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design manifest-set-top-level designCoverage '{"hasDesignSystem":true,"hasBrief":{current_hasBrief},"hasWireframes":...}'
```

**Why it is wrong:** `workflows/brief.md` anti-pattern section explicitly states "NEVER add a `hasBrief` flag to design-manifest.json." The live `design-manifest.json` confirms: `designCoverage` has keys `hasDesignSystem`, `hasWireframes`, `hasFlows`, `hasHardwareSpec`, `hasCritique`, `hasHandoff` — no `hasBrief`. The field cannot ever be set, so the preserve instruction is permanently dead and confusing.

**Also confirmed in flows.md:** `workflows/flows.md` Step 7 (lines 481, 491) also references `hasBrief` in the coverage-merge instruction, meaning the same fix applies to both workflow files.

**Fix for system.md Step 7:**
1. Remove the prose parenthetical about `hasBrief` from line 1326.
2. Remove `"hasBrief":{current_hasBrief}` from the `manifest-set-top-level` bash command on line 1330.

**Fix for flows.md Step 7:**
1. Remove `hasBrief` from the bullet list of current coverage values to preserve (line 481).
2. Remove `"hasBrief":{current}` from the `manifest-set-top-level` bash command (line 491).

**Current designCoverage schema (confirmed live):**
```json
{
  "hasDesignSystem": false,
  "hasWireframes": false,
  "hasFlows": false,
  "hasHardwareSpec": false,
  "hasCritique": false,
  "hasHandoff": false
}
```

The corrected coverage-merge commands must only reference these six keys.

### INT-02 + FLW-BREAK-02: `{count}` placeholder rows in Domain Files table

**Verified location:** Both `templates/design-state-root.md` (lines 47-52) and the live `.planning/design/DESIGN-STATE.md` (lines 47-52).

Both files contain:
```markdown
| strategy | strategy/DESIGN-STATE.md | {count} | 2026-03-15 |
| ux | ux/DESIGN-STATE.md | {count} | 2026-03-15 |
| visual | visual/DESIGN-STATE.md | {count} | 2026-03-15 |
| review | review/DESIGN-STATE.md | {count} | 2026-03-15 |
| hardware | hardware/DESIGN-STATE.md | {count} | 2026-03-15 |
| handoff | handoff/DESIGN-STATE.md | {count} | 2026-03-15 |
```

**Why it is wrong:** Domain files are created on demand when a skill first runs. Pre-populating rows with `{count}` placeholder implies those files exist — they don't. The skills' Step 6 logic appends rows to the Domain Files table as they create their domain DESIGN-STATE files. Pre-populated stubs cause the table to always look populated even when no skill has run.

**Audit fix options (from milestone audit):**
- Option A (preferred): Remove the pre-populated rows from the template. Let skills append on first run. The table comment already says "Domain files are created on demand."
- Option B: Replace `{count}` with `0` and date with `--`. Less clean but preserves skeleton structure.

**Chosen approach:** Option A — remove pre-populated rows from BOTH the template (`templates/design-state-root.md`) AND the live file (`.planning/design/DESIGN-STATE.md`). The comment block explaining the pattern is kept. This is cleaner and matches the intent of the existing comment.

**Fix:** Edit the Domain Files table in both files. Change from pre-populated rows to an empty table (header + comments only).

**Note:** The `ensureDesignDirs` function in `bin/lib/design.cjs` is NOT involved here — it only creates directories, not the DESIGN-STATE.md file. The template is used when /pde:brief runs `design ensure-dirs` and the `cmdEnsureDirs` path creates the root DESIGN-STATE. The live file was already created, so both must be fixed independently.

### INT-03: `tokens-to-css` CLI orphaned — system.md Step 5 uses inline Write

**Verified location:** `workflows/system.md` Step 5, File 10: `assets/tokens.css` (line ~1047).

Current behavior: Claude is instructed to write all CSS custom property declarations inline using the Write tool — a concatenation of all 7 category CSS files.

**The CLI exists and works:**
```bash
node bin/pde-tools.cjs design tokens-to-css <tokens-file>
```
It calls `generateCssVars(tokens)` which produces a `:root {}` block from a DTCG JSON file. The self-test verifies this: `design.cjs self-test: 19 tests — 19 passed, 0 failed`.

**Important constraint (verified from Phase 14 decision log):**
- `generateCssVars()` only wraps output in `:root {}` — it does NOT produce dark mode blocks
- `assets/tokens.css` must be inline (no `@import`) for `file://` URL compatibility
- Dark mode requires manual `@media + [data-theme]` blocks

**Scope of the fix:** The success criterion states "Step 5 calls `pde-tools.cjs design tokens-to-css` for CSS generation instead of inline Write." The practical fix is to update File 10 (`assets/tokens.css`) generation in system.md Step 5 to:
1. Bash-call `tokens-to-css` on `visual/SYS-tokens.json` to get the base `:root {}` block
2. Append dark mode blocks manually (as before — `generateCssVars` doesn't handle dark mode)

This wires the orphaned CLI into at least one real workflow call, satisfying INFRA-03 at the integration level.

**Critical: tokens-to-css output format:**
```bash
CSS=$(node "${CLAUDE_PLUGIN_ROOT}/bin/pde-tools.cjs" design tokens-to-css ".planning/design/visual/SYS-tokens.json" --raw)
```
The `--raw` flag returns the CSS string directly (not JSON-wrapped). The output is a single `:root { ... }` block.

## Architecture Patterns

### How Workflow Edits Work in This Project

All PDE workflows live in `workflows/*.md`. They are loaded by Claude via `@workflows/...` references in command files (`commands/*.md`). Edits to workflow files take effect immediately on next skill invocation — no build step, no compilation.

**Edit scope:**
- `workflows/system.md` — two edit sites (Step 7 hasBrief removal, Step 5 tokens-to-css wiring)
- `workflows/flows.md` — one edit site (Step 7 hasBrief removal)

### Self-Test Pattern (Established in Phases 12-13)

The design.cjs self-test pattern uses `--self-test` flag:
```bash
node bin/lib/design.cjs --self-test
```

Test groups use `check(description, fn)` helper. New tests append to an existing test group. No test framework dependency — pure Node.js assert + console.

**Adding a dark mode test (test group 3):**
```javascript
check('generateCssVars wraps in :root block (already exists)', ...);
// NEW:
check('dark mode block uses @media + [data-theme] pattern (manual assembly, not generateCssVars)', () => {
  // Verify generateCssVars does NOT produce @media blocks
  const css = generateCssVars({ color: { bg: { $value: '#fff', $type: 'color' } } });
  assert(!css.includes('@media'), 'generateCssVars must not produce @media blocks');
  assert(!css.includes('[data-theme]'), 'generateCssVars must not produce [data-theme] blocks');
});
```

## Common Pitfalls

### Pitfall 1: Removing hasBrief but leaving hasHardwareSpec
**What goes wrong:** `flows.md` Step 7 line 491 references both `hasBrief` AND `hasHardwareSpec`. The live manifest schema includes `hasHardwareSpec` (confirmed in design-manifest.json). Only `hasBrief` is invalid — `hasHardwareSpec` MUST stay.
**How to avoid:** Remove only `"hasBrief":{current}` from the coverage-merge command. Keep all other flags including `hasHardwareSpec`.

### Pitfall 2: Editing template but not live file (or vice versa) for INT-02
**What goes wrong:** Fix only `templates/design-state-root.md` but leave `{count}` in the live `.planning/design/DESIGN-STATE.md` — the live file still shows stale data.
**How to avoid:** Edit both files in the same task. Both are identified in the audit.

### Pitfall 3: tokens-to-css produces only :root{} — dark mode still needs manual assembly
**What goes wrong:** Assuming `tokens-to-css` handles the entire `assets/tokens.css` file. It produces ONE `:root {}` block from the DTCG JSON. The dark mode blocks (`@media (prefers-color-scheme: dark)` and `[data-theme="dark"]`) must still be written manually.
**How to avoid:** The Step 5 update instruction should: (a) call `tokens-to-css` for the base `:root {}`, (b) still include manual dark mode block instructions. The Phase 14 decision is documented: "generateCssVars() must NOT be used for dark mode."

### Pitfall 4: Flipping nyquist_compliant: true without adding the missing dark mode test
**What goes wrong:** Simply editing the frontmatter without addressing the Wave 0 item for dark mode tests — leaves test coverage gap.
**How to avoid:** Add the dark mode test to design.cjs self-test group 3, run the self-test to confirm passing, THEN flip the frontmatter.

### Pitfall 5: Skipping the live DESIGN-STATE.md BRF/SYS/FLW artifact rows
**What goes wrong:** The live `.planning/design/DESIGN-STATE.md` only has the Domain Files rows with `{count}` — it has NO artifact history for skills that have already run (BRF, SYS, FLW). After removing the `{count}` rows, the Domain Files section should reflect what is actually present.
**How to avoid:** After removing `{count}` rows from the live file, add current rows for domains that exist: `strategy` (BRF ran), `visual` (SYS ran), `ux` (FLW ran). Check which domain DESIGN-STATE.md files actually exist before adding rows.
