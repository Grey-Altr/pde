**workflows/competitive.md** — Create the full competitive skill workflow following the canonical structure from `workflows/brief.md`. The file MUST contain all 5 lint-required sections: `<purpose>`, `<skill_code>`, `<skill_domain>`, `<context_routing>`, `<process>`, plus `<required_reading>`, `<flags>`, and `<output>`.

Structure the workflow as follows:

1. **`<purpose>`**: Run structured competitive landscape analysis producing a CMP-competitive-v{N}.md artifact with competitor profiles, feature comparison matrix, SVG positioning map, Porter's Five Forces analysis, pricing analysis, gap analysis, and differentiation recommendations. Every factual claim carries a confidence label. The Opportunity Highlights section is the key downstream contract — consumed by /pde:opportunity as candidate input.

2. **`<skill_code>`**: CMP

3. **`<skill_domain>`**: strategy

4. **`<context_routing>`**: Detect PROJECT.md (hard requirement), existing BRF-brief artifact (soft — enriches competitor identification), REQUIREMENTS.md (soft), DESIGN-STATE (soft).

5. **`<required_reading>`**: Reference @references/skill-style-guide.md, @references/mcp-integration.md, @references/strategy-frameworks.md

6. **`<flags>`**: Support `--dry-run`, `--quick` (3 competitors, no Porter's), `--standard` (default: 3-5 competitors with Porter's), `--deep` (5-8+ competitors with additional maps), `--verbose`, `--no-mcp`, `--no-websearch`, `--no-sequential-thinking`, `--force`

7. **`<process>`**: Implement a 7-step pipeline:

   **Step 1/7: Initialize design directories** — `pde-tools.cjs design ensure-dirs` (exact pattern from brief.md Step 1)

   **Step 2/7: Check prerequisites and determine scope** — Read PROJECT.md (hard fail if missing). Optionally read BRF-brief-v*.md, REQUIREMENTS.md. Parse scope flag (--quick, --standard, --deep; default --standard). Check for existing CMP artifact and determine version N. If exists and no --force, prompt user. Determine competitor count based on scope.

   **Step 3/7: Probe MCP capabilities** — Check --no-mcp flag first. Then:
   - Probe WebSearch MCP: if available, use for live competitor data (pricing, features, funding). Tag verified claims as `[confirmed]`. If unavailable, set WEBSEARCH_AVAILABLE=false, tag `[Using training knowledge -- install WebSearch MCP for current market data]`, and default all training-knowledge claims to `[inferred]`
   - Probe Sequential Thinking MCP: if available, use for multi-step competitor analysis reasoning. If unavailable, continue without.

   **Step 4/7: Competitive analysis** — This is the core logic. Present a CONFIDENCE LABEL definition block early in this section that the executor must apply throughout:
   ```
   CONFIDENCE LABELS (apply to EVERY factual competitor claim):
   [confirmed]  — directly verified from competitor's own docs, pricing page, or WebSearch result
   [inferred]   — concluded from indirect evidence (category membership, common patterns)
   [unverified] — cannot confirm from available sources; flagged for user validation
   ```

   Analysis sections (scope determines which are included):
   - **Competitor identification**: Identify competitors based on product type, market, and features from PROJECT.md. Quick=3, Standard=3-5, Deep=5-8+.
   - **Competitor profiles**: For each competitor — name, URL, founding/funding, target market, key differentiators. All claims labeled.
   - **Feature comparison matrix**: Rows = key features from PROJECT.md/REQUIREMENTS.md. Columns = our product + competitors. Cells = full/partial/none with confidence label on each cell.
   - **SVG Positioning map**: Use the SVG template from references/strategy-frameworks.md. Pick meaningful axis pairs based on product domain (e.g., ease-of-use vs feature-depth, price vs capability). Place our product and each competitor as positioned dots.
   - **Porter's Five Forces** (skip if --quick): Use the framework from strategy-frameworks.md. Rate each force (Low/Medium/High) with rationale. Focus on forces most relevant to the product space.
   - **Pricing analysis**: Compare pricing tiers/models across competitors. Label each price point with confidence.
   - **Gap analysis**: Identify gaps where competitors are weak or absent. For each gap, note: which competitor comparison revealed it, estimated user segment affected, competitive advantage potential.
   - **Differentiation recommendations**: Strategic recommendations based on gaps and positioning.

   **Step 5/7: Write competitive artifact** — Use the template structure from templates/competitive-landscape.md EXACTLY. Write to `.planning/design/strategy/CMP-competitive-v{N}.md` with YAML frontmatter (Generated, Skill, Version, Status, Scope, Data Currency, Enhanced By). CRITICAL: The `## Opportunity Highlights` section MUST use the structured numbered-list format that /pde:opportunity will parse:
   ```markdown
   ## Opportunity Highlights
   1. **[Gap Name]**
      - Source: [Which competitor comparison revealed this]
      - Estimated reach: [user segment affected]
      - Competitive advantage: [why this gap matters]
   ```

   **Step 6/7: Update domain DESIGN-STATE** — Check for .planning/design/strategy/DESIGN-STATE.md; create from domain template if absent. Add/update CMP artifact row.

   **Step 7/7: Update root DESIGN-STATE and manifest** —
   - `lock-acquire pde-competitive`
   - Update root DESIGN-STATE: add/update cross-domain map entry, decision log entry, iteration history
   - `lock-release`
   - `manifest-update CMP code CMP`, `manifest-update CMP name "Competitive Landscape Analysis"`, `manifest-update CMP type competitive`, `manifest-update CMP domain strategy`, `manifest-update CMP path ".planning/design/strategy/CMP-competitive-v{N}.md"`, `manifest-update CMP status draft`, `manifest-update CMP version {N}`
   - Coverage flag: `coverage-check` to read all 13 flags, then `manifest-set-top-level designCoverage '{full 13-field JSON with hasCompetitive:true}'` using the pass-through-all pattern

8. **Summary table** and **Anti-Patterns** sections at the end of `<process>`:
   - Anti-patterns: never omit confidence labels from competitor claims, never hard-fail on WebSearch unavailability, never write Opportunity Highlights as prose (must be structured numbered list), never skip coverage-check before writing designCoverage, never use dot-notation with manifest-set-top-level, never skip Porter's in standard/deep mode

9. **`<output>`**: List files written (CMP-competitive-v{N}.md, DESIGN-STATE updates, manifest updates)
