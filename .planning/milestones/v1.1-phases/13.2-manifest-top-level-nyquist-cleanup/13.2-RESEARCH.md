## Summary

Phase 13.2 is a narrow, targeted gap-closure phase. It resolves two items of tech debt catalogued in the v1.1 milestone audit: (1) the `design-manifest.json` top-level fields `projectName` and `productType` are never populated after `/pde:brief` runs, and (2) the Phase 13.1 VALIDATION.md was created as a pre-execution draft and still carries `nyquist_compliant: false` despite the phase completing successfully with all tests passing.

The codebase is already mature and the patterns are established. The manifest gap requires adding one new function (`cmdManifestSetTopLevel`) to `bin/lib/design.cjs`, routing it through `bin/pde-tools.cjs` as the `manifest-set-top-level` subcommand, extending Step 7 of `workflows/brief.md` to call it, and adding a self-test assertion. The Nyquist gap requires updating two frontmatter fields and five checklist items in `13.1-VALIDATION.md`.

Both fixes are surgical. No new architecture is introduced. No dependencies change. The self-test count increments from 18 to 19 after adding the `manifest-set-top-level` assertion.

**Primary recommendation:** Implement `manifest-set-top-level` following the existing `cmd*` wrapper pattern in design.cjs, extend brief.md Step 7 with two calls (projectName, productType), then fix 13.1-VALIDATION.md frontmatter. All in one plan, two tasks.

### Pattern 1: cmd* Function in design.cjs

**What:** Every CLI-exposed operation is a thin `cmd*` wrapper. It validates args with `error()`, calls the underlying business function (`readManifest`/`writeManifest`), and calls `output(result, raw)`.

**When to use:** Any new subcommand exposed via pde-tools.cjs.

**Example (existing model — `cmdManifestUpdate`):**

```javascript
// Source: bin/lib/design.cjs lines 248-254
function cmdManifestUpdate(cwd, code, field, value, raw) {
  if (!code || !field) {
    error('code and field required for manifest-update');
  }
  const manifest = updateManifestArtifact(cwd, code, { [field]: value });
  output(manifest, raw);
}
```

**New function for 13.2 (`cmdManifestSetTopLevel`):**

```javascript
// Pattern mirrors cmdManifestUpdate — same guard, read, mutate, write, output shape
function cmdManifestSetTopLevel(cwd, field, value, raw) {
  if (!field) {
    error('field required for manifest-set-top-level');
  }
  const manifest = readManifest(cwd);
  if (!manifest) {
    error('design-manifest.json not found — run design ensure-dirs first');
  }
  manifest[field] = value;
  writeManifest(cwd, manifest);
  output(manifest, raw);
}
```

**Key constraint:** Only root-level scalar fields should be settable via this command. The function does `manifest[field] = value` — caller is responsible for passing the right field name. No schema validation needed (consistent with how `cmdManifestUpdate` trusts its `field` arg).

### Anti-Patterns to Avoid

- **Do not use `updateManifestArtifact` for root fields.** That function does `manifest.artifacts[code] = ...` — it cannot reach `manifest.projectName`. A new function is required.
- **Do not add a new `projectName` field to the `artifacts.BRF` sub-object.** The manifest schema has `projectName` at root, not nested. Do not create confusion between root metadata and artifact metadata.
- **Do not skip the lock when writing root manifest fields.** Step 7 already holds the write lock when calling `manifest-update`. The two new `manifest-set-top-level` calls happen inside the same lock window — no lock changes needed.
- **Do not validate the field name in `cmdManifestSetTopLevel`.** Consistent with `cmdManifestUpdate` pattern — no schema enforcement at the cmd layer. The workflow is the contract.
- **Do not change `outputRoot` or `mode` in this phase.** Those fields are set at manifest init time by `ensureDesignDirs`. Phase 13.2 only touches `projectName` and `productType`.

## Common Pitfalls

### Pitfall 1: Placing manifest-set-top-level calls AFTER lock-release

**What goes wrong:** `writeManifest` modifies the manifest file. If lock is released before the write completes, a concurrent writer could see a corrupt intermediate state.

**Why it happens:** The `lock-release` call is the last visual element in Step 7, so it's tempting to add new calls after it.

**How to avoid:** Insert the two new calls immediately after the last existing `manifest-update BRF version` call, before the `lock-release` block. The `lock-release` must remain the last write operation in Step 7.

**Warning signs:** Any `manifest-set-top-level` call outside the "Acquire write lock" / "ALWAYS release write lock" block.

### Pitfall 2: The `value` argument passes through as a raw string — no type coercion

**What goes wrong:** `cmdManifestSetTopLevel(cwd, 'productType', 'software', raw)` sets `manifest.productType = 'software'` which is correct. But `manifest.projectName` should be a human-readable name string. If the caller passes an empty string (because PROJECT.md has no name field), the manifest gets `projectName: ''`.

**Why it happens:** Bash command substitution returns empty string when the field doesn't exist. The function has no empty-string guard.

**How to avoid:** In `cmdManifestSetTopLevel`, add a guard:
```javascript
if (value === '' || value === undefined || value === null) {
  error('value cannot be empty for manifest-set-top-level');
}
```
Or alternatively, gracefully skip writing when value is empty by returning early with `output(manifest, raw)` without writing. The latter is more defensive for the `projectName` case.

**Warning signs:** `manifest.projectName` is `''` or `undefined` after running `/pde:brief`.

### Pitfall 3: VALIDATION.md status field not updated alongside Nyquist flags

**What goes wrong:** The frontmatter has `status: draft` (line 4). If only `nyquist_compliant` and `wave_0_complete` are updated, the document still reads as a draft even though the phase is complete.

**Why it happens:** The primary success criterion mentions only `nyquist_compliant: true`, which draws attention away from the `status` field.

**How to avoid:** Update `status: draft` → `status: complete` in the same edit. It's one field in the same frontmatter block.

**Warning signs:** Grep for `status: draft` in 13.1-VALIDATION.md after the fix.

### Pitfall 4: Self-test assertion for manifest-set-top-level misses the empty-string edge case

**What goes wrong:** The self-test only checks the happy path (valid field + value) but doesn't cover the case where `value` is empty.

**Why it happens:** Minimal test-per-function approach.

**How to avoid:** The happy path test is sufficient for 13.2 scope. Document the empty-string behavior in the function's JSDoc rather than adding a second test. The self-test count stays at 19 (one new assertion added).

### Self-test assertion (design.cjs runSelfTest, Test group 4)

```javascript
// Add after 'updateManifestArtifact creates artifact entry with code BRF' check
check('cmdManifestSetTopLevel sets root-level manifest field', () => {
  cmdManifestSetTopLevel(tmpDir, 'projectName', 'Test Project', false);
  const manifest = readManifest(tmpDir);
  assert.strictEqual(manifest.projectName, 'Test Project');
});
```

Note: The self-test uses `false` as the `raw` arg (same pattern as `cmdLockStatus` test). The `output()` call writes to stdout and calls `process.exit(0)` — requires the same process mock used in the `cmdLockStatus` test OR we use `readManifest` to verify the side-effect instead of capturing stdout. Reading back via `readManifest` is cleaner and avoids needing the mock.

Revised self-test assertion (no mock needed because we verify the file side-effect, not stdout):

```javascript
check('cmdManifestSetTopLevel sets root-level manifest field', () => {
  // Use process mock pattern consistent with cmdLockStatus test
  const origWrite = process.stdout.write;
  const origExit = process.exit;
  process.stdout.write = () => true;
  process.exit = () => {};
  cmdManifestSetTopLevel(tmpDir, 'projectName', 'Test Project', false);
  process.stdout.write = origWrite;
  process.exit = origExit;
  const manifest = readManifest(tmpDir);
  assert.strictEqual(manifest.projectName, 'Test Project');
});
```

## Open Questions

1. **How to source `projectName` from PROJECT.md**
   - What we know: PROJECT.md is loaded in Step 2 of brief.md. It may have a YAML frontmatter `name:` field, or the project name may only appear in headings/body text.
   - What's unclear: There is no standardized `name` field enforced by /pde:new-project. The PROJECT.md template may or may not include a top-level `name:` field.
   - Recommendation: Instruct the brief.md executor to use the project name as set in PROJECT.md's first `# Heading` OR the `name:` frontmatter field if present. This is a best-effort extraction, not a hard requirement. If extraction fails, the manifest should record `"unknown"` rather than fail.

2. **Should `outputRoot` and `mode` be populated in Phase 13.2?**
   - What we know: The manifest template has `outputRoot` and `mode` as placeholder strings. These were not mentioned in phase success criteria.
   - What's unclear: Whether downstream skills (Phase 14+) will need `outputRoot` before they can run.
   - Recommendation: Out of scope for 13.2. Only `projectName` and `productType` are in the success criteria. Defer `outputRoot` and `mode` to the first phase that needs them.
