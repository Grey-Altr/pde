**Step 4/7: Generate wireframe HTML per screen**

This is the core generation step. For EACH screen in SCREENS:

*4a. Determine screen context:*
- Screen slug, label, journey, persona, type (from inventory or argument)
- Product name, product type (from brief or PROJECT.md)
- Read the flow document (`FLW-flows-v*.md`) to understand the screen's role in the user journey (what comes before, what comes after, what triggers navigation to this screen).

*4b. Apply fidelity-specific content rules (STRICTLY — these are prescriptive, not guidelines):*

**Lo-fi rules (body class: `pde-layout pde-layout--lofi`):**
- Gray boxes with dimension labels for images/media: `[Image: 400x200]`
- Text labels ONLY: "Username", "Submit", "Navigation" — NO realistic content, NO lorem ipsum
- Dashed borders on ALL placeholder regions
- NO color from design tokens — neutral gray only (`#e5e7eb` backgrounds, `#6b7280` text, `#d1d5db` borders)
- Click-through `<a href>` links between pages ACTIVE
- Data tables: header row + 3 placeholder rows with gray cells, column count matches design
- Dashboards: all card regions as gray boxes with dimension + type annotation: `[Card: Metric -- 200x100]`
- Charts: bounding box only with chart type label: `[Chart: Line -- 400x200]`

**Mid-fi rules (body class: `pde-layout pde-layout--midfi`):**
- Real layout with proper spacing/alignment using token CSS custom property references (with fallback values)
- Realistic placeholder content: fake names, domain-relevant sample data (NOT lorem ipsum)
- SVG silhouette placeholders indicating content type (NOT icon fonts, NOT actual images)
- Basic chart outlines (bar/line/pie shapes, no data values)
- Primary state + empty state required
- Microcopy: realistic button labels ("Sign in", not "Submit"), validation messages, helper text
- Token variables referenced WITH fallback: `var(--space-4, 1rem)`, `var(--color-border, #e5e7eb)`

**Hi-fi rules (body class: `pde-layout pde-layout--hifi`):**
- Design tokens fully applied from assets/tokens.css — colors, typography, shadows, spacing, border-radius
- Near-real content from brief: product name, feature names, real messaging
- Inline SVG charts with sample data
- ALL state variants required: default + empty + loading + error
- Full microcopy with tooltips, helper text, validation messages
- Theme toggle button visible (calls `toggleTheme()` JS function)

*4c. Generate HTML using the exact scaffold from templates/wireframe-spec.md:*

```html
<!DOCTYPE html>
<html lang="en" data-theme="light">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{Product Name} - {Screen Label}</title>
  <link rel="stylesheet" href="../../assets/tokens.css">
  {If TOKENS_AVAILABLE is false: inline <style> block with fallback palette variables + comment}
  <style>
    @layer tokens, wireframe-layout, components, utilities;

    @layer wireframe-layout {
      /* Screen-specific layout styles following fidelity rules */
    }

    /* Responsive breakpoints: 640px, 768px, 1024px, 1280px */
  </style>
</head>
<body class="pde-layout pde-layout--{FIDELITY}">
  <a href="#main-content" class="skip-link">Skip to main content</a>
  <header role="banner">...</header>
  <nav aria-label="Main navigation">
    <a href="index.html">All Screens</a>
    {Inter-page links to ALL screens in INVENTORY, not just current batch}
    {For screens not yet generated: <!-- Not yet generated -- run /pde:wireframe "{slug}" {fidelity} to create this file -->}
  </nav>
  <main role="main" id="main-content">
    <!-- State: default -->
    <div class="pde-state pde-state--default" aria-live="polite">
      {Screen content following fidelity rules}
    </div>

    <!-- State: loading (hidden) -->
    <!-- ANNOTATION: {trigger description}. {skeleton/spinner description}. {success → transition}. {failure → error state}. -->
    <div class="pde-state pde-state--loading" hidden>
      {Loading skeleton or spinner}
    </div>

    <!-- State: error (hidden) -->
    <!-- ANNOTATION: {error trigger}. {error display}. {recovery action → retry/redirect}. -->
    <div class="pde-state pde-state--error" hidden>
      <div role="alert" class="pde-error-banner">{error message}</div>
      {Retry CTA}
    </div>
  </main>
  <footer role="contentinfo">...</footer>
  <script>
    /* Theme toggle (hi-fi only) */
    function toggleTheme() {
      const html = document.documentElement;
      html.setAttribute('data-theme', html.getAttribute('data-theme') === 'dark' ? 'light' : 'dark');
    }
  </script>
</body>
</html>
```

*4d. Annotation requirements (MANDATORY — Phase 19 handoff depends on these):*
- Every state variant block MUST have an `<!-- ANNOTATION: ... -->` comment describing:
  - What triggers this state
  - What the user sees (skeleton, spinner, error banner, etc.)
  - Recovery path (success transition, retry action, redirect)
- Every interactive element with non-obvious behavior MUST have an annotation:
  - Modals: `<!-- ANNOTATION: Click opens modal dialog with {content}. Close via X button or Escape key. -->`
  - Async forms: `<!-- ANNOTATION: Form submit triggers POST to /api/{resource}. Loading state replaces form. Success redirects to {screen}. -->`
  - Drawers/sidebars: `<!-- ANNOTATION: Click toggles sidebar visibility. Overlay dims main content. -->`
  - Dropdowns: `<!-- ANNOTATION: Click reveals dropdown menu with {N} options. Selection updates {target}. -->`

*4e. Accessibility requirements (ALL fidelity levels):*
- `role="banner"`, `role="main"`, `role="contentinfo"` landmarks
- `aria-label="Main navigation"` on nav
- Skip-to-main link at top of body: `<a href="#main-content" class="skip-link">Skip to main content</a>`
- All form inputs have `<label for="id">` associations
- All images have `alt` attributes (descriptive or `alt=""` for decorative)
- `aria-live="polite"` on state variant containers
- Logical tab order following visual layout
