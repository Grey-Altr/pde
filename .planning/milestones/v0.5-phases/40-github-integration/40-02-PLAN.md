**Create workflows/sync-github.md:**

Structure the workflow with these steps:

**Step 0 — Initialize and parse arguments:**
- Parse `--dry-run` flag from $ARGUMENTS
- Load repo info from mcp-connections.json via bridge:

```bash
node --input-type=module <<'EOF'
import { createRequire } from 'module';
const req = createRequire(import.meta.url);
const b = req(`${process.env.CLAUDE_PLUGIN_ROOT}/bin/lib/mcp-bridge.cjs`);
const conn = b.loadConnections();
const gh = conn.connections && conn.connections.github;
const repo = gh && gh.repo || '';
const status = gh && gh.status || 'not_configured';
const lookup = b.call('github:list-issues', {});
process.stdout.write(JSON.stringify({ repo, status, toolName: lookup.toolName }) + '\n');
EOF
```

If `repo` is empty or `status` is not `connected`, display degraded-mode message:
```
GitHub is not connected or no repo configured.
Run /pde:connect github to set up, then /pde:connect github --confirm to record your repo.
Sync skipped — no changes to REQUIREMENTS.md.
```
Stop here. Do NOT crash.

**Step 1 — Probe GitHub MCP availability:**
Use the probe gate pattern (from research Pattern 2). Call `b.probe('github')`. If probe returns `available: false`, display:
```
GitHub MCP server is not available. Run /pde:connect github to set up.
Sync will run in degraded mode — showing current REQUIREMENTS.md state without GitHub data.
```
Then display current `### GitHub Issues` section from REQUIREMENTS.md if it exists, and stop.

**Step 2 — Fetch GitHub issues via MCP:**
Split the `repo` value on `/` to get `owner` and `repoName`.

Instruct Claude to call the MCP tool whose name was returned by `bridge.call('github:list-issues', ...)` — i.e., `mcp__github__list_issues` — with these parameters:
- `owner`: the parsed owner
- `repo`: the parsed repoName
- `state`: `"OPEN"` (UPPERCASE — GraphQL enum, NOT lowercase)
- `perPage`: 50

Important: The actual MCP tool call happens in Claude Code's execution context (as a normal tool use), NOT inside a bash block. The workflow should instruct Claude to use the tool name from the bridge lookup.

Handle pagination: if the response includes `pageInfo.hasNextPage === true`, call `list_issues` again with `after: pageInfo.endCursor`. Cap at 4 pages (200 issues max). Display a message when cap is hit: "Showing first 200 open issues. Run /pde:sync --github again later to check for new issues."

**Step 3 — Deduplicate against existing REQUIREMENTS.md:**
Read `.planning/REQUIREMENTS.md`. For each fetched issue, check if `#<issue_number>` already appears anywhere in the file (followed by a non-digit character or end of line). If it does, skip that issue (already synced). Collect new issues that are NOT yet present.

**Step 4 — Append new issues to REQUIREMENTS.md:**
If `--dry-run`: display what would be appended but do not write. Show count and list.

If NOT dry-run:
- Find or create a `### GitHub Issues` section in REQUIREMENTS.md. If the section does not exist, append it after the last existing section (before the `---` footer if present).
- Append each new issue in this format:
  ```
  - [ ] **GH-<number>**: <issue title> ([#<number>](https://github.com/<owner>/<repo>/issues/<number>))
  ```
- Write the updated REQUIREMENTS.md.

**Step 5 — Display summary:**
```
GitHub Sync Complete (<owner>/<repo>)

  Issues fetched: <total>
  Already synced:  <skipped>
  Newly added:     <added>

New requirements added to .planning/REQUIREMENTS.md under ### GitHub Issues.
```

If no new issues: "All GitHub issues already synced. No changes to REQUIREMENTS.md."

**Error handling:** If any MCP tool call fails, display the error message and fall back to degraded mode (show current state without GitHub data). Never crash.

**Important workflow conventions:**
- All bash blocks use `node --input-type=module` with `createRequire()` (Phase 39 pattern — required by posttooluse-validate hook)
- Never call raw MCP tool names directly in workflow text — always look up via `bridge.call('github:list-issues', args)` first to get the toolName

**Step 2 — Format and display:**
Display a header:


For each workflow run in the response, display one row:
```
  <event>/<head_branch>    <workflow_name>    <status>    <conclusion or -->    <relative_time>
```

Where:
- `event`: e.g., "push", "pull_request"
- `head_branch`: branch that triggered the run
- `workflow_name`: name of the workflow (e.g., "CI")
- `status`: "queued", "in_progress", "completed"
- `conclusion`: "success", "failure", "cancelled", etc. — show `--` if status is not "completed"
- `relative_time`: compute from `created_at` ISO timestamp (e.g., "2 min ago", "1 hr ago", "running now" for in_progress)

Status values to handle: `queued`, `in_progress`, `completed`, `requested`, `waiting`.
Conclusion values when completed: `success`, `failure`, `cancelled`, `skipped`, `timed_out`.

After the table, display a footer:
```
Run /pde:pipeline-status to refresh.
```

**Error handling:** If the MCP tool call fails, display:
