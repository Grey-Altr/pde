# Phase 40.1: GitHub Integration Tech Debt Cleanup - Research

**Researched:** 2026-03-18
**Domain:** Claude Code command frontmatter (allowed-tools) + workflow defensive error handling
**Confidence:** HIGH

## Summary

Phase 40.1 closes four specific consistency gaps identified by the v0.5 milestone audit. Two gaps are in command frontmatter: `commands/sync.md` and `commands/pipeline-status.md` are missing `mcp__github__*` in their `allowed-tools` lists, which blocks Claude Code from calling GitHub MCP tools at runtime even though their downstream workflows require those tools. Two gaps are in workflow bash blocks: `workflows/brief-from-github.md` and `workflows/handoff-create-prs.md` call `b.call()` without try/catch, unlike the established defensive pattern in `workflows/sync-github.md`.

All four changes are surgical one-to-five line edits. There are no architectural decisions to make — the correct patterns are already established in sibling files. The planner's job is to apply those patterns exactly. No new libraries, no new files, no secondary effects.

**Primary recommendation:** Apply the two reference patterns verbatim. Do not introduce variations.

## Architecture Patterns

### Pattern 1: Command allowed-tools with MCP wildcard

**What:** Claude Code command files use YAML frontmatter to declare which tools Claude is permitted to call. The `allowed-tools` list must include `mcp__github__*` (wildcard) for any command that delegates to a workflow making GitHub MCP tool calls — even if those calls happen in a sub-workflow rather than directly in the command file itself.

**Reference file:** `commands/brief.md` (lines 7-13) and `commands/handoff.md` (lines 7-15)

**Exact format from `commands/brief.md`:**
```yaml
---
name: pde:brief
description: Generate a structured product design brief
argument-hint: "[--from-github <issue-url-or-number>] [--quick] [--dry-run] [--verbose] [--no-mcp] [--force]"
allowed-tools:
  - Read
  - Write
  - Edit
  - Bash
  - Glob
  - Grep
  - Task
  - mcp__github__*
---
```

**Key rule:** The wildcard `mcp__github__*` is a single list entry. It is not quoted. It appears after all standard tools. The order is: Read, Write, Edit, Bash, Glob, Grep, Task (if present), then MCP wildcards.

**For `commands/sync.md`:** Add `mcp__github__*` after `Grep` (no `Edit`, no `Task` in this command).

**For `commands/pipeline-status.md`:** Add `mcp__github__*` after `Bash` (minimal tool list — only Read and Bash currently).

### Pattern 2: Defensive try/catch around b.call() in bash blocks

**What:** Workflow bash blocks that look up a tool name via `b.call()` must wrap that call in try/catch. This ensures that if the TOOL_MAP lookup fails (e.g., key not found, bridge error), the bash block still emits valid JSON with an empty `toolName` rather than throwing an unhandled exception.

**Reference file:** `workflows/sync-github.md` Step 0 bash block (lines 14-32)

**Exact pattern from `workflows/sync-github.md`:**
```javascript
let toolName = '';
try {
  const lookup = b.call('github:list-issues', {});
  toolName = lookup.toolName;
} catch (err) {
  toolName = '';
}
```

**Key rules:**
- Initialize `toolName` to `''` before the try block
- Inside try: assign `b.call(...)` to `lookup`, then read `lookup.toolName`
- Inside catch: reassign `toolName = ''` (do not rethrow)
- The catch parameter name is `err`
- The `process.stdout.write` call is OUTSIDE the try/catch, using the `toolName` variable

**For `workflows/brief-from-github.md`:** The Step 1 bash block currently has a bare `const lookup = b.call('github:get-issue', {});` that is not try/catch wrapped. Apply the pattern, keeping the same canonical key `'github:get-issue'`.

**For `workflows/handoff-create-prs.md`:** The Step 0 bash block currently has a bare `const lookup = b.call('github:create-pr', {});` that is not try/catch wrapped. Apply the pattern, keeping the same canonical key `'github:create-pr'`.

### Anti-Patterns to Avoid

- **Do not change the canonical key strings** (`'github:get-issue'`, `'github:create-pr'`). These map to TOOL_MAP entries in mcp-bridge.cjs. Changing them breaks the adapter layer.
- **Do not change the surrounding bash block structure** — only wrap the `b.call()` line. The `loadConnections()` call, `process.stdout.write()`, and other logic in the same bash block are correct and must not be touched.
- **Do not add `mcp__github__*` to commands that do not delegate to GitHub workflows** — the pattern is scoped to commands that actually invoke GitHub MCP tools through their workflow chain.
- **Do not quote the wildcard** in YAML frontmatter. Use `  - mcp__github__*` not `  - "mcp__github__*"`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead |
|---------|-------------|-------------|
| Determining correct tool names for MCP calls | Custom lookup logic | `b.call(canonicalKey, {})` from mcp-bridge.cjs — already implemented |
| Detecting GitHub availability | Custom availability checks | `b.probe('github')` — already implemented in sync-github.md |
| Error message formatting | Custom degraded-mode messages | Follow the existing degraded-mode messages already in each workflow |

**Key insight:** All infrastructure for graceful degradation is already in place. Phase 40.1 only adds the standard defensive wrapper to the one place in each workflow that lacks it.

## Common Pitfalls

### Pitfall 1: Touching the wrong bash block in brief-from-github.md

**What goes wrong:** The file has two bash blocks. Step 1 has the `b.call()` that needs wrapping. Step 0 only does argument parsing (no `b.call()`). Editing the wrong block introduces errors.

**Why it happens:** Scanning too fast and not reading the full step context.

**How to avoid:** The bash block requiring the fix is in **Step 1 — Load repo info and check GitHub availability** (line 48). It contains `const lookup = b.call('github:get-issue', {});` on approximately line 56. Do not edit the Step 0 section.

**Warning signs:** If you see `parseInt` or URL parsing logic in the block being edited, you are in the wrong section.

### Pitfall 2: Breaking the process.stdout.write call

**What goes wrong:** The `toolName` variable used in `process.stdout.write(JSON.stringify({ repo, status, toolName }) + '\n')` must remain in scope. If try/catch is applied incorrectly (e.g., wrapping the entire block including `process.stdout.write`), variable scope can break.

**Why it happens:** Wrapping too much of the block in try/catch.

**How to avoid:** Only the `b.call()` assignment needs try/catch. The `let toolName = ''` declaration, the `loadConnections()` call, and the `process.stdout.write` all live OUTSIDE the try/catch.

**Warning signs:** If the refactored bash block has `process.stdout.write` inside the try block, it is wrong.

### Pitfall 3: Wrong YAML indentation in allowed-tools

**What goes wrong:** YAML list items require exactly two spaces of indentation before the `-`. Using four spaces or mismatching the existing list items will cause frontmatter parse errors.

**Why it happens:** Copy-paste from a context that uses different indentation.

**How to avoid:** Match the indentation of existing entries in the same file. Both target command files use two-space-indented list items: `  - Read`.

**Warning signs:** If the added line does not visually align with `  - Read` and `  - Bash` in the file, the indentation is wrong.

### Pitfall 4: Omitting mcp__github__* from pipeline-status.md

**What goes wrong:** `commands/pipeline-status.md` has only `Read` and `Bash` in its allowed-tools. It is easy to overlook because the command file is very short and the MCP call is buried inside `workflows/pipeline-status.md`.

**Why it happens:** The command file looks complete — it delegates entirely to the workflow, so there is no visible indication that MCP tools are needed.

**How to avoid:** The audit specifically identified this file. Add `mcp__github__*` as the third entry in the allowed-tools list.
