# Pitfalls Research

**Domain:** Adding self-improvement capabilities, skill builder, tool optimization, Awwwards-level design elevation, and end-to-end pressure testing to an existing AI development platform (PDE v1.3)
**Researched:** 2026-03-17
**Confidence:** HIGH (grounded in direct PDE codebase inspection, existing architecture decisions in PROJECT.md, and verified against research on self-improving AI, meta-prompting, AI design quality, and LLM testing patterns)

---

## Critical Pitfalls

### Pitfall 1: Self-Improvement Fleet Audits Its Own Output, Creating Confirmation Loops

**What goes wrong:**
The self-improvement fleet consists of agents that audit, validate, and elevate PDE's output quality. If those agents evaluate their own work — or evaluate work produced by agents trained on the same prompts they are improving — they produce circular confirmation rather than genuine critique. An agent that generates a skill prompt and then audits it for quality will consistently rate it highly, because both the generation and the evaluation are anchored to the same implicit quality model. The fleet "passes" everything it ships. Quality regressions slip through invisibly. The system declares itself excellent while producing output that an external judge would rate mediocre.

**Why it happens:**
It is architecturally simple to use the same Claude instance for both generation and evaluation. The path of least resistance is: generate skill → ask Claude "is this skill high quality?" → Claude says yes → mark as improved. Developers do not immediately see the failure because outputs look plausible and internally consistent.

**How to avoid:**
Separate the generation context from the evaluation context with structural independence. The audit agent must evaluate against an external fixed rubric — not against the generating agent's own intent. The rubric must be written by a human before the audit agent exists, not generated by the system. For skill quality auditing specifically: define the rubric in a reference file (e.g., `references/skill-quality-rubric.md`) that the audit agent reads as ground truth, never modifies, and cannot override. The audit agent returns a structured verdict against that rubric, not a free-text quality rating. For design elevation: use Awwwards-specific criteria (typography hierarchy, motion intentionality, spatial rhythm, visual distinctiveness) as fixed rubric entries — not "is this design good?" but "does this design meet criterion X?"

**Warning signs:**
- Self-improvement fleet consistently rates its own skill rewrites as "significantly improved" with no failures
- Audit agents accept generated prompts that violate documented constraints (e.g., missing required fields from skill-registry.md)
- The rubric used for evaluation is itself AI-generated rather than human-authored

**Phase to address:** The tool audit framework phase — before any self-improvement agents are built, establish the external fixed rubric they will evaluate against. This rubric is a human-authored artifact, not a system output.

### Pitfall 2: Skill Builder Generates Skills That Break the Existing Plugin Architecture

**What goes wrong:**
A skill builder capable of creating and modifying PDE skills must produce output that conforms to the Claude Code plugin format: correct frontmatter schema (name, description, argument-hint, allowed-tools), valid workflow delegation (`Follow @workflows/X.md exactly`), correct skill registry registration, and template references that resolve. If the skill builder generates syntactically plausible but structurally invalid skill files, the plugin loads without error but the new skill either does nothing, crashes mid-execution, or produces output that orphans from the planning state. Claude Code does not validate skill files at install time — broken skills fail silently at invocation.

**Why it happens:**
The skill builder is prompted to "create a skill like the existing ones" but is not given the exact format constraints as executable validation criteria. It pattern-matches to the visible structure but misses non-obvious requirements: the `allowed-tools` list must contain only tools Claude Code actually exposes; the workflow file must exist at the referenced path; the skill code in `skill-registry.md` must be unique. Without automated format validation as a post-generation step, these invisible contract violations slip through.

**How to avoid:**
The skill builder must run a validation pass after generating any skill file, before presenting it to the user. The validation must check: (1) frontmatter parses as valid YAML; (2) `allowed-tools` entries are all from the known-valid Claude Code tool list; (3) the referenced `@workflows/` path exists or is being created in the same operation; (4) the skill code (BRF, SYS, etc.) is unique in `skill-registry.md`; (5) if a template is referenced, the template file exists in `templates/`. Implement this as a `pde-tools.cjs validate-skill {path}` command that the skill builder calls automatically. Reject the skill and retry if validation fails — never present a skill that fails validation as complete.

**Warning signs:**
- Skill builder produces `.md` files with `allowed-tools` entries that don't exist in Claude Code (e.g., `- Write` when the command only needs `Read`)
- Generated workflow files reference `@templates/X.md` paths that don't exist
- Two generated skills share the same skill code in `skill-registry.md`
- Running a generated skill produces no output (silent failure from invalid workflow delegation)

**Phase to address:** Skill builder implementation phase — the validation command must be built before the skill builder ships. Skill builder without validation is a broken-skills generator.

### Pitfall 3: Skill Upgrades Break Existing User Projects Mid-Pipeline

**What goes wrong:**
The self-improvement fleet upgrades existing PDE skills — rewriting prompts, adjusting workflow logic, updating template references. Users who have an in-progress project (partially complete pipeline, DESIGN-STATE.md tracking coverage flags, artifacts on disk) run an upgraded skill that expects a different artifact schema, different coverage flag names, or different frontmatter fields in existing artifacts. The skill fails against their existing state, or worse, silently produces an artifact that is incompatible with the next stage downstream. The user cannot resume their pipeline without manual intervention.

**Why it happens:**
Skill upgrades are tested against fresh projects. Existing-project compatibility is not tested because it requires understanding what state a partially-complete pipeline might be in. The self-improvement fleet "validates" by checking whether a skill runs successfully on a test project, not whether it degrades gracefully against all valid in-progress states.

**How to avoid:**
Before any skill upgrade ships, run a backward-compatibility check: (1) Create a minimal snapshot of a partially-complete project state (DESIGN-STATE.md with some flags true, some false; a mix of old-format artifacts in `.planning/design/`); (2) Run the upgraded skill against this snapshot; (3) Verify it either succeeds or fails with a human-readable error, never silently corrupts state. Document the compatibility contract for each skill in a machine-readable format: minimum required artifacts, minimum required coverage flag state, artifact schema version. When a skill upgrade changes the contract, it must also ship a migration guide. Require that skill upgrades include a "compatibility delta" comment in the workflow file header.

**Warning signs:**
- Self-improvement fleet only tests skills against empty projects or freshly initialized planning state
- Skill rewrites change artifact frontmatter field names without a migration path
- Running an upgraded skill on a project with v1.1-format artifacts produces a new artifact that references fields absent from v1.1 artifacts

**Phase to address:** Self-improvement fleet design phase — compatibility testing harness must be defined before the fleet begins modifying existing skills.

### Pitfall 4: Design Elevation Produces "AI Aesthetic" — Technically Correct but Distinctively Generic

**What goes wrong:**
The Awwwards-level design elevation effort upgrades the design pipeline output. The most likely failure mode is not broken output — it is visually coherent, technically valid output that is instantly recognizable as AI-generated. The symptoms: hero sections with oversized geometric sans-serifs in a single weight; gradients that drift from purple-to-teal; spacing that is mathematically proportional but rhythmically inert; motion that is smooth but purposeless (fade-in, slide-up, no choreography); layouts that are responsive but compositionally timid. This output scores well on automated metrics (contrast ratios, spacing consistency, token compliance) but would be eliminated in the first round at Awwwards because it has no point of view.

**Why it happens:**
LLMs trained on web design are trained on the modal web — the average of millions of sites, weighted toward common patterns. When asked to produce "professional, high-quality design," the LLM produces the statistical center of the training distribution: Tailwind defaults, Inter or Geist, card grids, hero + CTA. This is competent but undifferentiated. Awwwards winners are at the extreme tail of the distribution, not the center.

**How to avoid:**
The design elevation prompts must not ask for "high quality design" or "professional design." They must specify exact named criteria that distinguish Awwwards winners from the median: (1) **Typography** — specify that the type system must use at least two typefaces with intentional contrast (not just size contrast), with a named rationale for the pairing; (2) **Motion** — specify that animations must be choreographed to content meaning, not decorative; enter animations are triggered by scroll context; (3) **Spatial rhythm** — whitespace must be deliberate and asymmetric in at least one axis; symmetric grid everywhere is a failure signal; (4) **Visual distinctiveness** — require a "visual hook" — one design decision that would read as intentional and memorable in isolation. Build these criteria into the design elevation rubric as pass/fail gates, not soft suggestions. The critique and iterate skills must evaluate against these specific criteria, not against "does it look good."

**Warning signs:**
- Mockup output uses Inter or Geist as the sole typeface with no rationale for the choice
- All animations in the mockup are opacity or transform translations with identical easing curves
- The hero section layout is identical across projects (large text left, CTA button below, background gradient)
- Spacing uses consistent 8px or 16px multiples throughout with no deliberate breaks in the rhythm

**Phase to address:** Design quality elevation phase — the Awwwards criteria rubric must be authored before any elevation work begins. The rubric is input to the work, not an output of it.

### Pitfall 5: Pressure Test Measures the Wrong Things — Artifacts, Not Outcomes

**What goes wrong:**
The end-to-end pressure test runs the full 13-stage pipeline on a real project and measures whether each stage completes. Coverage flags are all `true`. Artifacts exist. The pipeline "passed." But the actual quality of the output — whether the brief is strategically coherent, whether the design tokens produce a distinctive visual language, whether the wireframes reflect real interaction patterns, whether the mockup is genuinely Awwwards-quality — is not measured. The pressure test is a completeness check masquerading as a quality check. PDE ships v1.3 with a "validated" pipeline that produces mediocre output at scale.

**Why it happens:**
Completeness is measurable automatically. Coverage flags are boolean. Artifact existence is verifiable with a file glob. Quality is harder to measure — it requires a rubric and human judgment. Pressure test authors default to what they can automate, which is process compliance rather than outcome quality.

**How to avoid:**
The pressure test must include two distinct phases: (1) **Process compliance check** — all stages complete, all coverage flags set, all artifacts exist at expected paths, artifact frontmatter is valid, no orphaned files; (2) **Outcome quality check** — a human (or structured AI-with-rubric) evaluates each artifact against domain-specific quality criteria. The quality check must use the same Awwwards rubric defined in Pitfall 4. Specifically: the mockup is evaluated against typography, motion, spatial rhythm, and visual distinctiveness criteria; the brief is evaluated for strategic coherence (does the market positioning follow logically from the competitive analysis?); the handoff is evaluated for engineering completeness (are all component APIs specified, are all design tokens mapped to CSS variables?). Pressure test is only "passed" when both phases pass. Document the quality check results in a structured report, not just a boolean.

**Warning signs:**
- Pressure test completion criteria are expressed entirely as boolean flags (hasIdeation, hasMockup, etc.)
- No human review step is planned for the pressure test
- "Pressure test passed" is determined by running a coverage-check command and seeing all true

**Phase to address:** Pressure test phase design — before execution, specify both the process compliance criteria AND the quality evaluation criteria. Quality criteria must be written first; they drive what the pressure test is testing.

### Pitfall 6: Tool Audit Conflates Tool Availability with Tool Effectiveness

**What goes wrong:**
The tool audit framework checks MCP integrations, CLIs, agent prompts, and templates. The likely failure mode: the audit confirms that tools are installed and reachable, but does not evaluate whether they are producing the quality of output they are capable of. Context7 is reachable → audit passes. But the agent prompts using Context7 may be querying it with vague queries ("how does React work?") rather than precise queries ("React 19 useOptimistic hook API signature and return type"). MCP tools are available but the agent prompts use 10% of their capability. The audit reports "all tools operational" when the actual problem is that the tools are being used poorly.

**Why it happens:**
Availability is easy to measure (ping the tool, get a response). Quality of use requires understanding what the tool is capable of and evaluating whether the agent prompts achieve that capability. Audit authors default to availability because effectiveness requires domain expertise.

**How to avoid:**
The tool audit must define "effectiveness criteria" for each tool, not just "is it reachable." For Context7: effectiveness means queries are library-specific with version context, not generic; the audit runs a representative query and evaluates whether the response contains version-specific API details. For agent prompts: effectiveness means the prompt instructs the agent to use the tool for its unique capability (e.g., Context7 for library docs, not for general knowledge questions that training data handles). For templates: effectiveness means the template produces output that a senior practitioner in the domain would recognize as professionally complete. Define effectiveness criteria before the audit runs, test against those criteria, and rate each tool "effective / partially effective / ineffective" — not just "available / unavailable."

**Warning signs:**
- Tool audit checklist only asks whether tools respond to a ping
- Context7 usage in agent prompts uses broad queries rather than precise library-version-specific queries
- Agent prompts use WebSearch for questions that Context7 would answer with higher accuracy
- Audit report has no "effectiveness" dimension — only availability

**Phase to address:** Tool audit framework phase — effectiveness criteria must be defined before the audit runs. Without them, the audit measures the wrong dimension.

### Pitfall 7: Meta-Prompt Drift — Skill Builder's Own Prompts Degrade Over Iterative Self-Modification

**What goes wrong:**
The skill builder modifies PDE skills and can potentially modify workflow files that govern how skills are structured. If the skill builder is itself a workflow file, and the self-improvement fleet can modify workflow files, then the skill builder may improve itself over time — shortening its own instructions, removing constraints it finds verbose, simplifying its own output format. Each individual modification looks like an improvement. After five iterations, the skill builder has optimized away the constraints that prevented it from generating broken skills. The output degrades progressively, with each version slightly worse but passing self-evaluation because the rubric has also drifted.

**Why it happens:**
This is the AI alignment tax applied to a file-based system. Without a frozen ground truth, iterative self-modification has no stable attractor. The system optimizes for "internally consistent" rather than "externally valid." Self-evaluation cannot detect degradation in the rubric used for evaluation.

**How to avoid:**
The skill builder's core workflow file must be protected from self-modification. Mark it explicitly as a "human-only edit" file — the self-improvement fleet's modification scope must exclude it. The way to enforce this in a file-based system: maintain a `protected-files.json` list in the plugin root that the self-improvement fleet reads before modifying any file; if a file is on the protected list, the fleet skips it and flags it for manual review. The skill quality rubric (`references/skill-quality-rubric.md`) is also on the protected list. These files can only be modified by a human commit — not by a generated PR from an improvement agent.

**Warning signs:**
- The skill builder workflow file has been modified by an improvement agent (check git blame)
- The skill quality rubric has fewer constraints in the current version than in the version from three improvement cycles ago
- Generated skills pass the quality rubric but fail manual review (rubric drift)

**Phase to address:** Self-improvement fleet design phase — define and implement the protected-files mechanism before the fleet gains write access to any workflow files.

### Pitfall 8: Design Quality Measurement Creates False Confidence Through Automated Metrics

**What goes wrong:**
The design elevation workflow adds automated quality checks: contrast ratios (WCAG compliance), spacing consistency (8px grid adherence), token coverage (all colors from design system). These checks all pass. The team concludes the design quality is high and ships. The actual problem: WCAG compliance does not measure visual interest; spacing grid compliance does not measure compositional intentionality; token coverage does not measure whether the token choices are aesthetically coherent. Automated metrics measure technical correctness, not design quality. A site can score 100% on every automated check and look like a generic SaaS template.

**Why it happens:**
Automated metrics are objectively measurable and unambiguous. Design quality is subjective and requires judgment. Teams reach for automated metrics because they remove the need for qualitative evaluation. The automation produces a number, the number looks good, the team ships.

**How to avoid:**
Establish a two-tier measurement system in the design elevation workflow. Tier 1 (automated): technical compliance checks (contrast, spacing tokens, responsive breakpoints, animation timing functions within acceptable ranges). Tier 2 (rubric-based): qualitative evaluation against the Awwwards criteria rubric — evaluated by an AI agent reading the rubric and producing a structured verdict, not a score, against each criterion. The tier 2 evaluation must produce specific, actionable findings ("the type pairing uses two weights of the same typeface — this fails the typeface contrast criterion; introduce a serif display face for headings") not a numeric score. A design is only cleared for the pressure test if both tiers are satisfied. Numeric scores from tier 1 are not proxies for tier 2 quality.

**Warning signs:**
- Design elevation is declared complete after WCAG and spacing checks pass with no qualitative evaluation
- The critique and iterate skills are not updated to evaluate against Awwwards criteria
- Quality reports contain only numeric scores, no specific design findings

**Phase to address:** Design quality elevation phase AND the critique/iterate skill update phase — both must implement the two-tier measurement before any quality evaluation runs.

### Pitfall 9: Pressure Test Is Run Only on a Greenfield Project, Missing In-Progress State Failures

**What goes wrong:**
The pressure test runs the full 13-stage pipeline from scratch on a new project. Everything passes. The test is declared successful. But the most common real-world user scenario is resuming a partially-complete pipeline: `/pde:build --from wireframe` after critique found issues, or re-running `/pde:system` after updating a brief. These resume paths are not tested. Users encounter failures in the resume flows — the paths PDE v1.2 actually introduced — while the greenfield path that was tested continues to work.

**Why it happens:**
It is simpler to test the happy path from the beginning. Resume and re-run paths require constructing specific mid-pipeline states, which requires test fixtures. Developers skip fixture construction in the interest of shipping the pressure test quickly.

**How to avoid:**
The pressure test must include at minimum three distinct pipeline entry states: (1) greenfield (no planning state); (2) partially complete (DESIGN-STATE.md with 5-8 stages complete, artifacts on disk for those stages); (3) re-run (a stage that has already run, being re-run after a manual artifact edit). The `--from` flag specifically must be tested with: valid stage name (proceeds correctly), invalid stage name (fails with helpful error listing valid stages), and a stage name that is already complete (user confirmation prompt, not silent re-execution). Test fixture construction is a required deliverable of the pressure test phase, not optional prep work.

**Warning signs:**
- Pressure test plan contains only "run /pde:build from scratch" scenarios
- The `--from` flag is tested only with the happy path (valid stage name, stage not yet complete)
- No test fixture exists representing a partially-complete project

**Phase to address:** Pressure test phase — test fixtures must be defined and built as the first deliverable, before any test execution.

### Pitfall 10: Awwwards Elevation Applies to Mockup Only, Missing System and Wireframe Stages

**What goes wrong:**
The design elevation effort focuses on the mockup skill — the visible, hi-fi output. The system (design tokens) and wireframe (structural layout) stages are not elevated. The result: the design system generates a token set with a color palette derived from brand heuristics rather than intentional color theory; the wireframe layout uses safe 12-column grid patterns without considering spatial tension or hierarchy. When mockup generates hi-fi from these inputs, it is constrained by the low quality of its upstream sources. The mockup cannot achieve Awwwards quality if the token system it consumes is generic and the wireframe structure it interprets is templated.

**Why it happens:**
Mockup is the most visible output — it is what users see and evaluate. System and wireframe are perceived as functional steps whose quality is secondary. Elevation effort concentrates where quality is most visible rather than where quality is most foundational.

**How to avoid:**
Design quality elevation must be pipeline-aware. Work backward from Awwwards criteria to identify which stage is responsible for each quality dimension: (1) **Typography system** → `/pde:system` (type scale, pairing rationale, weight contrast) must produce a token set that encodes typographic ambition, not just sizes; (2) **Spatial rhythm** → `/pde:wireframe` must produce wireframes with deliberate compositional decisions documented as annotations (why this layout, what tension it creates); (3) **Visual distinctiveness** → `/pde:system` (color system) must generate a palette with a defined personality, not just accessible color pairs; (4) **Motion choreography** → `/pde:mockup` (this is appropriately at mockup level). Elevation phases must follow this order: system → wireframe → critique/iterate (to evaluate elevated inputs) → mockup. Elevating mockup without elevating system first is building quality on a weak foundation.

**Warning signs:**
- Design elevation work plan shows `/pde:mockup` as the only skill being upgraded
- The system skill continues to use a generic OKLCH palette generation heuristic with no typographic rationale
- Wireframe skill produces layouts without compositional annotations explaining spatial decisions

**Phase to address:** Design quality elevation phase — phase must begin with system skill elevation before touching mockup.

### Pitfall 11: Self-Improvement Agents Write to the Same Files They Are Reading, Causing Race Conditions in Multi-Agent Runs

**What goes wrong:**
The self-improvement fleet consists of multiple agents running in parallel or in sequence. If two agents in the fleet both read `design-manifest.json`, both intend to update it, and both write back their version, the second write overwrites the first agent's changes. In a sequential fleet, this is a lost update. In a parallel fleet (if PDE ever runs multiple agents concurrently), this is a race condition. The manifest ends up in an inconsistent state: coverage flags that were set by one agent are missing from the final manifest because another agent overwrote them.

**Why it happens:**
The file-based state model has no locking mechanism. Each agent reads, modifies, and writes independently. This is a known limitation of PDE's architecture (no database, no server) and was managed in v1.1/v1.2 by the pass-through-all coverage pattern. The self-improvement fleet introduces a new class of writer (agents that are themselves improving the system) that may not respect the same conventions as design skills.

**How to avoid:**
The self-improvement fleet must follow the same pass-through-all coverage pattern established in v1.2. More critically: self-improvement agents must not write to manifest files, coverage flags, or DESIGN-STATE.md as a side effect of their improvement work. The only file an improvement agent may write is the target skill/workflow/template file it is improving. If it needs to record improvement metadata, it writes to a dedicated `improvements/` log directory, never to shared planning state files. Run the fleet strictly sequentially (one agent at a time, one file at a time) — no parallelism until a locking mechanism exists.

**Warning signs:**
- Self-improvement agents modify `design-manifest.json` as part of their improvement operation
- Fleet runs multiple agents in parallel against the same planning state directory
- Coverage flags set early in a fleet run are absent from the manifest after the fleet completes

**Phase to address:** Self-improvement fleet design phase — the write scope constraint must be defined before any agent in the fleet is implemented.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Using the same Claude invocation to generate and evaluate a skill | Simpler implementation | Circular confirmation; self-improvement declares success without genuine validation | Never — generation and evaluation must use separate context anchors |
| Audit framework that only checks tool availability (ping test) | Fast, automatable | Tools rated "operational" while being used at 10% of their capability | Never — effectiveness criteria required alongside availability |
| Pressure testing only the greenfield happy path | Simpler test execution | Resume and re-run paths untested; most real-world failures occur there | Never for v1.3 validation; acceptable as a preliminary smoke test only |
| Applying Awwwards elevation to mockup only, not system/wireframe | Visible quality improvement faster | Mockup quality ceiling is set by system/wireframe quality; downstream quality cannot exceed upstream quality | Never — system must be elevated first |
| Allowing self-improvement agents to write to any file they can access | Maximum flexibility | Manifest corruption; coverage flag loss; breaking changes to shared state | Never — agent write scope must be explicitly constrained |
| Generating the skill quality rubric from AI output | Faster to ship | Rubric optimized for what AI can already do, not for what good looks like | Never — rubric is a human artifact |
| Skipping backward-compatibility testing on skill upgrades | Faster upgrade cycle | Users with in-progress projects cannot resume; support burden increases | Acceptable only if the upgrade is additive and adds no new required artifact fields |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Self-improvement fleet → existing skill workflows | Fleet modifies a skill workflow without running validation against the skill registry schema | All workflow modifications must pass `pde-tools.cjs validate-skill` before being written |
| Skill builder → Claude Code plugin format | Generated skills reference `@workflows/` paths that don't exist yet | Skill builder must create the workflow file in the same operation as the command file, or validate path existence before accepting the skill as complete |
| Design elevation → critique/iterate skills | Mockup is elevated but critique and iterate still evaluate against v1.1 criteria | Critique and iterate must be updated with Awwwards rubric criteria before design elevation is tested end-to-end |
| Tool audit → Context7 effectiveness | Audit checks Context7 is reachable but not whether agent prompts query it effectively | Audit must include a representative query execution and evaluate response quality |
| Pressure test → DESIGN-STATE.md | Pressure test creates planning state but DESIGN-STATE.md format has drifted since v1.1 | Verify DESIGN-STATE.md format matches what every skill expects before running pressure test |
| Skill builder → skill-registry.md | Skill builder generates a skill code that collides with an existing code | Skill builder must read skill-registry.md and validate code uniqueness before generating |
| Self-improvement fleet → protected files | Improvement agent modifies skill-quality-rubric.md, causing rubric drift | `protected-files.json` must be enforced before the fleet has write access to any file |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Running the full 13-stage pressure test in a single Claude invocation | Context exhaustion after stage 6-7; later stages produce shallow output due to compressed context | Run pressure test stage-by-stage; each stage is its own invocation with fresh context; aggregate results in a report file | Any run that requires more than ~8 design artifact generations in sequence |
| Self-improvement fleet processing all skills in a single pass | Later skills in the pass receive compressed context; improvements are shallower and more template-like | Process skills in batches of 3-4; write improvement log between batches; resume from log | When the fleet processes more than 5 skills per invocation |
| Design elevation generating full mockup HTML with motion for all screens at once | Incomplete HTML; animation keyframes truncated; design token application inconsistent | Generate mockup one screen at a time; include motion specification in a separate animation manifest | When mockup includes more than 2 screens or any CSS animation sequences |
| Tool audit scanning all workflow files in one read operation for quality assessment | Shallow assessment on later files; earlier files receive more analytical attention | Process audit in domain batches (strategy skills, design skills, review skills); write findings between batches | When audit scope exceeds 6-8 workflow files |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Skill builder with unrestricted write access to all plugin files | An improvement agent could modify the plugin's core configuration (CLAUDE.md, bin scripts, package.json), breaking the plugin for all users | Scope skill builder write access to `commands/`, `workflows/`, and `templates/` only; bin scripts and config files are explicitly excluded |
| Self-improvement agents reading `.env` files or user API keys from project context | Training data contamination or accidental key exposure in generated artifacts | Fleet agents must not have file read access outside `.planning/`, `commands/`, `workflows/`, `templates/`, `references/` |
| Pressure test running on a real user project rather than a synthetic test fixture | Overwrites real project artifacts; corrupts real project planning state | Pressure test must initialize its own isolated planning state in a test directory, never in an active user project's `.planning/` |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Skill builder outputs a generated skill without showing the user what was created | User does not know what they got; cannot evaluate quality before use | Skill builder must always display the generated skill content and ask for user confirmation before writing to disk |
| Self-improvement fleet runs silently with no progress indication | User cannot tell if the fleet is running or hung; no visibility into what is being improved | Fleet must emit a progress line for each skill it processes ("Auditing /pde:system... found 2 issues → rewriting") |
| Design elevation upgrades all design skills simultaneously | If one upgraded skill introduces a regression, the user cannot isolate which skill caused it | Elevate one skill at a time; verify each elevation with a standalone test before proceeding to the next |
| Pressure test failure report is a wall of text | User cannot identify what failed or where to start fixing | Structure pressure test output as a per-stage status table with pass/fail and a specific finding for each failure; failures sorted by severity |
| Tool audit recommends replacing a tool the user has already customized | User loses their customizations when they apply the recommendation | Audit must check for user-modified files (via git diff or modification date) and flag customizations before recommending replacement |

## "Looks Done But Isn't" Checklist

- [ ] **Self-improvement rubric:** `references/skill-quality-rubric.md` exists, is human-authored, and is listed in `protected-files.json` — verify it was not generated by an AI agent
- [ ] **Skill validation command:** `node bin/pde-tools.cjs validate-skill commands/X.md` runs without error on any generated skill before it is presented to the user — verify by running it on a known-broken skill and confirming rejection
- [ ] **Backward compatibility:** Running an upgraded skill against a v1.1-format project state (DESIGN-STATE.md with 7-flag coverage) does not corrupt the existing coverage flags — verify with a v1.1 state fixture
- [ ] **Awwwards rubric criteria:** Critique and iterate workflow files have been updated to evaluate against the Awwwards rubric, not just against "design quality" — verify by checking that the critique workflow references the rubric file by path
- [ ] **Protected files enforcement:** The self-improvement fleet, when processing a file in `protected-files.json`, skips it and logs the skip rather than modifying it — verify by adding a test file to the protected list and running the fleet
- [ ] **Pressure test resume paths:** The pressure test plan includes at least one `--from stage` scenario and one re-run scenario, not just the greenfield path — verify by checking the test plan document
- [ ] **Design system elevation:** The `/pde:system` workflow has been updated with Awwwards typography and color criteria BEFORE the mockup elevation work begins — verify by checking git commit order
- [ ] **Tool effectiveness criteria:** The tool audit framework has documented effectiveness criteria for each tool (Context7, Brave Search, agent prompts) before any audit runs — verify by checking the audit rubric for non-availability metrics
- [ ] **Fleet write scope:** The self-improvement fleet has an explicit list of directories it is permitted to write to, and that list excludes `bin/`, the plugin root config files, and `.planning/` shared state files — verify by checking the fleet's configuration or workflow constraints
- [ ] **Pressure test fixtures:** At minimum two test fixtures exist representing distinct pipeline states (greenfield and partially-complete) before the pressure test executes — verify by checking fixture files in the test setup

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Circular self-evaluation (fleet rates its own output as high quality) | HIGH | Author external rubric from scratch (human effort); re-run evaluation against rubric; expect many previously "passing" skills to fail — treat failures as the real baseline |
| Skill builder produced invalid skills already in production | MEDIUM | Run `validate-skill` against all generated skills; identify failures; delete or manually fix invalid skills; add validation to the skill builder before any further generation |
| Skill upgrade broke in-progress user projects | HIGH | Revert the skill upgrade via git; document the compatibility breakage in a migration guide; re-ship with a version flag (`--compat v1.2`) that enables legacy behavior |
| Design elevation produced generic AI aesthetic | MEDIUM | Delete elevation output; author the Awwwards rubric that was missing; re-run elevation with rubric as hard constraint; expect multiple critique/iterate cycles before passing |
| Pressure test declared success on process compliance only | LOW | Re-define pressure test criteria to include quality evaluation tier; re-run with human evaluation pass; document quality findings as pressure test artifacts |
| Self-improvement agent wrote to shared manifest files, corrupting state | MEDIUM | Restore manifest from git; re-run affected design stages; constrain fleet write scope before next fleet run |
| Rubric drift from iterative self-modification | HIGH | Restore skill-quality-rubric.md from the version before drift began (git history); diff to find what was removed; add to protected-files.json; manually review all skills improved under the drifted rubric |

## Pitfall-to-Phase Mapping

| Pitfall | Prevention Phase | Verification |
|---------|------------------|--------------|
| Circular self-evaluation (Pitfall 1) | Tool audit framework phase — before any self-improvement agents are built | Audit agents produce failures on deliberately-low-quality test skills; not all-pass |
| Skill builder generates invalid skills (Pitfall 2) | Skill builder implementation phase — validation command required before builder ships | Running `validate-skill` on a generated skill catches known-invalid patterns |
| Skill upgrades break in-progress projects (Pitfall 3) | Self-improvement fleet design phase — compatibility harness defined first | Upgraded skills tested against v1.1-format state fixtures without corruption |
| AI aesthetic / generic output (Pitfall 4) | Design quality elevation phase — Awwwards rubric authored first | Critique produces specific typography/motion/spatial failures on unrevised mockups |
| Pressure test measures completeness not quality (Pitfall 5) | Pressure test design phase — quality evaluation criteria defined before execution | Pressure test plan includes explicit human-review or rubric-evaluation pass |
| Tool audit measures availability not effectiveness (Pitfall 6) | Tool audit framework phase — effectiveness criteria defined in audit rubric | Audit produces "partially effective" or "ineffective" ratings, not just "available" |
| Meta-prompt / skill builder drift (Pitfall 7) | Self-improvement fleet design phase — protected-files mechanism implemented | Fleet skips protected files and logs skips rather than modifying them |
| False confidence from automated design metrics (Pitfall 8) | Design quality elevation phase AND critique/iterate update phase | Quality reports contain specific qualitative findings, not only numeric scores |
| Pressure test misses resume/re-run paths (Pitfall 9) | Pressure test design phase — test fixtures built as first deliverable | Test plan includes `--from` and re-run scenarios with passing results |
| Elevation applied to mockup only, not system/wireframe (Pitfall 10) | Design quality elevation phase — phase ordered system → wireframe → mockup | System skill produces type pairing with documented rationale before mockup elevation begins |
| Fleet race conditions on shared files (Pitfall 11) | Self-improvement fleet design phase — write scope constraints defined first | Fleet run log shows sequential processing; manifest has no lost-update artifacts |
