## Phase Details

### Phase 29: Quality Infrastructure
**Goal**: The measurable quality bar and protection mechanisms exist before any agent, audit, or elevated skill runs
**Depends on**: Nothing (first phase of v1.3)
**Requirements**: QUAL-01, QUAL-02, QUAL-03, QUAL-04, QUAL-05, QUAL-06
**Success Criteria** (what must be TRUE):
  1. `references/quality-standards.md` exists with Awwwards 4-dimension scoring (Design 40%, Usability 30%, Creativity 20%, Content 10%) and concrete criteria per score level 1-10
  2. `references/motion-design.md` exists with animation timing scales, named easing curves, scroll-driven effect techniques, GSAP 3.14 CDN patterns, and variable font axis animation examples
  3. `references/composition-typography.md` exists with named grid systems, visual weight analysis, type pairing contrast rationale, spatial asymmetry principles, and APCA contrast guidance
  4. `protected-files.json` exists and lists quality rubric, core workflow files, and bin/ scripts — with an explicit allowed-write-directories list
  5. All three new agent types (AUD, IMP, PRT) are registered in `bin/lib/model-profiles.cjs` and `skill-registry.md`
**Plans**: 3 plans

Plans:
- [ ] 29-01-PLAN.md — Awwwards quality rubric reference (QUAL-01)
- [ ] 29-02-PLAN.md — Motion design and composition/typography references (QUAL-02, QUAL-03)
- [ ] 29-03-PLAN.md — Protected-files mechanism, model profiles, skill registry (QUAL-04, QUAL-05, QUAL-06)

### Phase 30: Self-Improvement Fleet & Audit Command
**Goal**: PDE can evaluate its own output quality against the Awwwards rubric and generate grounded improvement proposals
**Depends on**: Phase 29 (quality rubric and protected-files must exist before any fleet agent runs)
**Requirements**: AUDIT-01, AUDIT-02, AUDIT-03, AUDIT-04, AUDIT-05, AUDIT-06, AUDIT-07, AUDIT-08, AUDIT-09, AUDIT-10, AUDIT-11, AUDIT-12
**Success Criteria** (what must be TRUE):
  1. Running `/pde:audit` produces `.planning/audit-report.md` with severity-rated findings (CRITICAL/HIGH/MEDIUM/LOW) across commands, agents, templates, references, and agent prompts
  2. The auditor agent evaluates artifacts in read-only mode against `references/quality-standards.md` — it never writes to any file outside `.planning/improvements/`
  3. The improver/validator loop can take an audit finding, propose a fix with before/after diff, and apply it to PDE files with the protected-files guard enforced
  4. `pde-tools validate-skill` CLI command exists and checks frontmatter YAML validity, allowed-tools correctness, workflow path existence, and skill code uniqueness
  5. `/pde:audit` produces a PDE Health Report with before/after baseline measurements — running it twice against the same state yields a quantifiable delta, not a re-statement of findings
**Plans**: 3 plans

Plans:
- [ ] 30-01-PLAN.md — validate-skill CLI, fleet agent definitions, model profiles (AUDIT-02, AUDIT-03, AUDIT-04, AUDIT-08, AUDIT-12)
- [ ] 30-02-PLAN.md — /pde:audit command and workflow with 3-agent fleet orchestration (AUDIT-01, AUDIT-05, AUDIT-06, AUDIT-07)
- [ ] 30-03-PLAN.md — Baseline/delta tracking, health report, Nyquist tests (AUDIT-09, AUDIT-10, AUDIT-11)

### Phase 31: Skill Builder
**Goal**: PDE can create and improve skills that conform to the plugin format without producing broken or pre-elevation-quality output
**Depends on**: Phase 29 (quality references), Phase 30 (validate-skill CLI delivered here, audit baseline exists)
**Requirements**: SKILL-01, SKILL-02, SKILL-03, SKILL-04, SKILL-05, SKILL-06
**Success Criteria** (what must be TRUE):
  1. Running `/pde:improve create "description"` produces a valid SKILL.md that passes `pde-tools validate-skill` automatically — the command rejects invalid output before presenting it to the user
  2. Running `/pde:improve improve skill-name` produces targeted enhancements additive by default, with full rewrite requiring `--rewrite` flag — existing skills are never silently overwritten
  3. Running `/pde:improve eval skill-name` produces a quality score with specific findings mapped to rubric dimensions — not generic qualitative commentary
  4. Skill builder writes to `commands/` and user project `.claude/skills/` — it cannot write to `bin/`, plugin root config, or any protected path
  5. Skill builder reads `references/skill-style-guide.md` and `references/tooling-patterns.md` as constraint references — generated skills match the 7-step anatomy of existing elevated skills
**Plans**: 2 plans

Plans:
- [ ] 31-01-PLAN.md — Agents, rubric reference, model-profiles, Nyquist tests (SKILL-03, SKILL-04, SKILL-05, SKILL-06)
- [ ] 31-02-PLAN.md — /pde:improve command and three-mode workflow orchestrator (SKILL-01, SKILL-02, SKILL-03, SKILL-04, SKILL-05, SKILL-06)

### Phase 32: Design Elevation — System Skill
**Goal**: The system skill produces motion-aware, perceptually harmonious design tokens that set a high ceiling for all downstream design work
**Depends on**: Phase 29 (quality references must exist for injection), Phase 30 (audit baseline before elevation begins)
**Requirements**: SYS-01, SYS-02, SYS-03, SYS-04, SYS-05, SYS-06
**Success Criteria** (what must be TRUE):
  1. `/pde:system` output includes DTCG-format motion tokens: a duration scale with 5+ steps (micro to dramatic), named easing curves (ease-in, ease-out, ease-in-out, spring, bounce), and delay tokens for choreography sequencing
  2. `/pde:system` output includes variable font axis animation tokens: weight range, width range, optical size, and recommended animation parameters for each axis
  3. `/pde:system` color palette output uses OKLCH with documented perceptual harmony rationale (analogous, complementary, split-complementary, or triadic) — not arbitrary color selection
  4. `/pde:system` output includes APCA-aware contrast guidance with Lc values at each type scale step — not just WCAG ratio compliance
  5. `/pde:system` output includes type pairing recommendations with documented contrast rationale (typeface category contrast, not just size contrast)
**Plans**: TBD

### Phase 33: Design Elevation — Wireframe Skill
**Goal**: Wireframe output demonstrates intentional composition decisions that can be traced to specific design principles — not default symmetrical layouts
**Depends on**: Phase 32 (system skill must be elevated first — wireframe quality ceiling is set by upstream tokens)
**Requirements**: WIRE-01, WIRE-02, WIRE-03, WIRE-04, WIRE-05
**Success Criteria** (what must be TRUE):
  1. `/pde:wireframe` output names the grid system used (12-column, modular, golden ratio, or asymmetric) with documented rationale for that choice per layout — not an unnamed default grid
  2. `/pde:wireframe` output annotates visual weight distribution per layout: identifies where the eye is drawn first and why, not just what content appears where
  3. `/pde:wireframe` output intentionally breaks symmetry on at least one axis per page with a documented purpose — accidental or default symmetry is flagged in the annotation
  4. `/pde:wireframe` output shows distinct layout strategies for mobile (375), tablet (768), and desktop (1440) — not the same layout fluid-scaled across breakpoints
  5. `/pde:wireframe` output numbers content priority (1st, 2nd, 3rd attention) per viewport — users can read the intended reading order from the annotation layer
**Plans**: TBD

### Phase 34: Design Elevation — Critique & HIG Skills
**Goal**: Critique evaluates against the Awwwards rubric specifically and flags AI aesthetic patterns; HIG audits motion accessibility with the same precision
**Depends on**: Phase 29 (quality rubric must exist for critique to evaluate against), Phase 33 (wireframe elevation — critique runs after wireframe in the pipeline order)
**Requirements**: CRIT-01, CRIT-02, CRIT-03, CRIT-04, HIG-01, HIG-02, HIG-03
**Success Criteria** (what must be TRUE):
  1. `/pde:critique` output maps every finding to one of the four Awwwards dimensions (Design, Usability, Creativity, Content) with a specific observation — not a generic checklist item
  2. `/pde:critique` explicitly identifies AI aesthetic patterns by name when present: generic gradients, symmetric grids, template-like interactions, stock typography — each flagged item includes a specific remediation instruction
  3. `/pde:critique` assesses motion choreography: states whether motion is purposeful and narrative or decorative and random, with specific examples from the artifact
  4. `/pde:hig` motion accessibility audit identifies prefers-reduced-motion violations and names vestibular-safe alternatives for any parallax or scroll effect in the artifact
  5. `/pde:hig` animation performance check identifies any animations causing layout reflow and confirms GPU-composited properties are used where claimed — specific element names are cited, not generic warnings
**Plans**: TBD

### Phase 35: Design Elevation — Mockup Skill
**Goal**: Mockup output produces concept-specific, choreographed interactions and animations that would not be mistaken for generic AI-generated design
**Depends on**: Phase 32 (system tokens), Phase 33 (wireframe layout quality), Phase 34 (critique framework — critique/iterate runs before mockup in pipeline order)
**Requirements**: MOCK-01, MOCK-02, MOCK-03, MOCK-04, MOCK-05, MOCK-06, MOCK-07
**Success Criteria** (what must be TRUE):
  1. `/pde:mockup` output uses spring physics animations (cubic-bezier approximations or GSAP CDN) for interactive elements — no linear or generic ease-only transitions on interactive controls
  2. `/pde:mockup` output includes scroll-driven animations using `@scroll-timeline` with `@supports` progressive enhancement fallback — statically readable without scroll support
  3. Every interactive element in `/pde:mockup` output has visually and kinetically distinct states: default, hover, focus, active, loading, disabled, error — all seven states are present, not aliased together
  4. `/pde:mockup` output contains at least one named "visual hook" — a concept-specific interaction or visual element unique to the project, identified by name in the artifact
  5. `/pde:mockup` entrance animations follow narrative content order — elements appear in the sequence they are meant to be read, not all-at-once or random stagger
  6. `/pde:mockup` uses variable font features: weight animation on hover, optical size adjustment by context, or width shifts for emphasis — documented in the artifact
**Plans**: TBD

### Phase 36: Design Elevation — Handoff, Flows & Cross-Cutting
**Goal**: All remaining pipeline skills are elevated, the reference injection pattern is applied uniformly, and the elevation delta is verified against the pre-elevation audit baseline
**Depends on**: Phase 32 (system), Phase 33 (wireframe), Phase 34 (critique/HIG), Phase 35 (mockup) — all upstream elevation must be complete before cross-cutting verification runs
**Requirements**: HAND-01, HAND-02, FLOW-01, CROSS-01, CROSS-02, ORDER-01
**Success Criteria** (what must be TRUE):
  1. `/pde:handoff` TypeScript interface output includes motion specifications per component: expected animation behavior (duration, easing, trigger) documented as interface fields or JSDoc annotations — not omitted from the interface
  2. `/pde:handoff` output includes implementation notes for any concept-specific interactions identified in the mockup (e.g., "this hero uses scroll-driven parallax — recommend GSAP ScrollTrigger or CSS @scroll-timeline")
  3. `/pde:flows` output annotates screen-to-screen transitions with the visual mechanism (slide, fade, morph, shared-element) — not just which screen follows which
  4. All elevated skills (`system.md`, `wireframe.md`, `mockup.md`, `critique.md`, `hig.md`, `handoff.md`, `flows.md`) load new quality references via `@` includes in `required_reading` — no structural changes to the 7-step skill anatomy
  5. Running `/pde:audit` after all elevation phases produces a measurable quality delta against the Phase 30 baseline — audit findings in elevated skill areas are fewer or lower severity than before
**Plans**: TBD

### Phase 37: Pressure Test & Validation
**Goal**: The full 13-stage pipeline runs on a real product concept and passes both process compliance and Awwwards-rubric quality evaluation
**Depends on**: Phases 29-36 (all infrastructure, fleet, elevation, and skill builder must be complete before final validation)
**Requirements**: PRES-01, PRES-02, PRES-03, PRES-04, PRES-05, PRES-06
**Success Criteria** (what must be TRUE):
  1. `/pde:pressure-test` command exists and runs the full 13-stage pipeline on a real product concept (not PDE itself) — producing `.planning/pressure-test-report.md`
  2. The pressure test process compliance tier passes: all 13 coverage flags set, design-manifest.json complete, artifacts exist at expected paths for all stages
  3. The pressure test quality rubric tier produces specific design findings per artifact against the Awwwards 4-dimension rubric — findings cite named elements, not generic observations
  4. The pressure test explicitly evaluates for AI aesthetic avoidance: concept-specific interactions present, non-generic color choices documented, intentional asymmetry visible, custom motion choreography identified — each as a named pass/fail check
  5. The pressure test runs successfully from three fixture states: greenfield (no prior artifacts), partially-complete (5-8 stages done), and re-run of a completed stage — all three produce a complete report without errors
