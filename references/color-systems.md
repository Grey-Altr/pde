## OKLCH Color Space Fundamentals

### What OKLCH Is

OKLCH is a perceptually uniform cylindrical color space with three channels:

| Channel | Range | Description |
|---|---|---|
| **L** (Lightness) | 0 - 1 | 0 = black, 1 = white. Perceptually linear -- L=0.5 actually looks mid-brightness |
| **C** (Chroma) | 0 - ~0.4 | Color intensity/saturation. 0 = achromatic gray, higher = more vivid |
| **H** (Hue) | 0 - 360 | Color wheel angle. 0/360 = pink-red, 90 = yellow, 180 = cyan, 270 = blue |

CSS syntax: `oklch(0.55 0.18 250)` -- Lightness Chroma Hue (no commas, space-separated)

### Why OKLCH Over HSL

**HSL's critical flaw: non-uniform lightness perception.**

In HSL, `hsl(60, 100%, 50%)` (yellow) and `hsl(240, 100%, 50%)` (blue) both have L=50%, but yellow appears vastly brighter than blue to the human eye. This makes algorithmic palette generation unreliable -- a 10-step blue scale and a 10-step yellow scale at the same HSL lightness values will look completely different in perceived brightness.

**OKLCH solves this.** At the same L value, all hues appear equally bright. This means:
- Palette scales generated algorithmically look consistent across all hues
- Step 500 in blue and step 500 in yellow actually appear the same brightness
- Dark mode palettes can be generated by lightness inversion with predictable results
- Color harmony rules produce visually balanced combinations

### Browser Support

The `oklch()` CSS function is supported in all modern browsers:

| Browser | Version | Release Date |
|---|---|---|
| Chrome | 111+ | March 2023 |
| Firefox | 113+ | May 2023 |
| Safari | 15.4+ | March 2022 |
| Edge | 111+ | March 2023 |

**Usage:** Safe to use natively. No polyfill needed for modern browser targets.

### Gamut Considerations

OKLCH can express colors outside the sRGB gamut (the standard range for web displays). High-chroma values at certain hue/lightness combinations may exceed what sRGB monitors can display.

**Handling strategy:**
1. **During palette generation:** Clamp chroma to the maximum sRGB-safe value for each hue/lightness combination
2. **CSS behavior:** Modern browsers using `oklch()` automatically perform gamut mapping (clamping) for out-of-gamut colors
3. **Hex fallbacks:** Generate hex color equivalents for each OKLCH value. These serve as fallbacks for any edge case rendering and as human-readable references
4. **Safe chroma ranges:** At L=0.5 (mid-brightness), most hues are safe up to C=0.20-0.25. At extreme L values (very light/dark), safe chroma is lower (C=0.05-0.15)

## Palette Generation Algorithm

### Overview

Design system palettes are generated from a small number of seed values. The algorithm takes a brand color and produces a full 11-step scale (50 through 950) suitable for UI design.

### Primary Palette Generation

**Input:** User provides a brand primary color (any format: hex, rgb, hsl, oklch)

**Process:**
1. Convert input color to OKLCH
2. Extract the Hue (H) value -- this stays fixed across the entire scale
3. Map the input color's Lightness to the 500 step
4. Generate all 11 steps using the scale table below

### Scale Generation Table

For each step, apply the Lightness (L) value directly and multiply the input color's Chroma (C) by the multiplier:

| Step | Lightness (L) | Chroma Multiplier | Character |
|---|---|---|---|
| 50 | 0.97 | 0.08 | Near-white tint, barely colored |
| 100 | 0.93 | 0.18 | Very light tint, subtle color |
| 200 | 0.87 | 0.32 | Light tint, noticeable color |
| 300 | 0.77 | 0.55 | Medium-light, clear color identity |
| 400 | 0.67 | 0.80 | Approaching full saturation |
| 500 | 0.55 | 1.00 | Full color -- primary brand shade |
| 600 | 0.47 | 0.90 | Slightly darker, good for hover states |
| 700 | 0.39 | 0.75 | Dark, high contrast on light backgrounds |
| 800 | 0.31 | 0.55 | Very dark, text-suitable on light backgrounds |
| 900 | 0.23 | 0.35 | Near-black shade with color tint |
| 950 | 0.15 | 0.20 | Darkest usable shade |

**Formula per step:**
```
L_step = table value (fixed)
C_step = C_input * multiplier (from table)
H_step = H_input (fixed -- hue never changes)
Result: oklch(L_step C_step H_step)
```

### Fixed Hue Principle

**The hue (H) value stays constant across all steps in a palette.** Only Lightness and Chroma vary. This ensures:
- The entire scale reads as "shades of the same color"
- No unexpected hue drift at light or dark extremes
- Predictable results when combining steps from different palettes

### Neutral Palette Generation

Neutrals use the same algorithm with near-zero chroma to produce tinted grays:

```
Neutral palette:
  C = 0.005 - 0.015 (barely perceptible tint)
  H = brand primary hue (creates warm/cool gray that harmonizes with brand)
  L = same scale table as primary
```

**Example:** Brand primary at H=250 (blue) produces cool-tinted neutrals. Brand primary at H=30 (orange) produces warm-tinted neutrals.

For pure gray (no tint), use C=0 with H=0.

### Semantic Color Derivation

Semantic colors (success, warning, error, info) use fixed hue ranges with the brand's lightness distribution:

| Semantic | Hue (H) | Typical Range | Rationale |
|---|---|---|---|
| Success | ~145 | 130-160 | Green -- universal "positive" association |
| Warning | ~85 | 70-100 | Amber/yellow -- caution without alarm |
| Error | ~25 | 15-35 | Red -- universal "danger/problem" signal |
| Info | ~250 | 235-265 | Blue -- neutral informational tone |

Generate each semantic color as a full 11-step palette using the same scale generation table. This ensures consistent lightness behavior across all palettes.

### Secondary and Accent Palette Generation

If the user provides a secondary color, generate its palette the same way as primary.

If no secondary is provided, auto-generate using color harmony rules from the primary hue:

| Strategy | Hue Offset | When to Use |
|---|---|---|
| Complementary | +180 degrees | High contrast accent, use sparingly |
| Analogous | +30 or -30 degrees | Cohesive, low-contrast companion |
| Triadic | +120 or -120 degrees | Balanced variety, good for multi-color systems |
| Split-complementary | +150 and -150 degrees | High contrast with more nuance than complementary |

**Default recommendation:** Analogous (+30 degrees) for secondary, as it produces the most cohesive design.

## Color Harmony Rules

### Complementary (180 degrees apart)

```
Primary: H
Complement: H + 180
```

- Highest contrast between two hues
- Creates strong visual tension
- Best for: accent buttons, call-to-action elements, error vs success differentiation
- Caution: using both at full saturation is visually aggressive -- use one at high chroma, other at lower

### Analogous (30 degrees apart)

```
Primary: H
Analogous 1: H + 30
Analogous 2: H - 30
```

- Low contrast, inherently harmonious
- Creates a cohesive, unified feel
- Best for: backgrounds, surfaces, secondary UI elements
- Works well for gradient combinations

### Triadic (120 degrees apart)

```
Primary: H
Triadic 1: H + 120
Triadic 2: H - 120 (or H + 240)
```

- Balanced variety across the color wheel
- Each color is equally visually prominent
- Best for: data visualization (3 distinct categories), multi-brand systems
- Works best when one color dominates and others accent

### Split-Complementary (150 degrees apart)

```
Primary: H
Split 1: H + 150
Split 2: H - 150 (or H + 210)
```

- High contrast with more flexibility than pure complementary
- Less visual tension while maintaining emphasis
- Best for: hero sections, feature highlights, marketing layouts

## Dark Mode Palette Generation

### Core Principle: Not Simple Inversion

Dark mode is NOT created by inverting lightness values. Simple inversion produces:
- Washed-out colors that lack vibrancy
- Incorrect contrast ratios
- Uncomfortable reading experiences

Instead, dark mode requires **independent generation with verified contrast**.

### Approach: Semantic Mapping Swap

The primitive palette (the 11-step scales) stays identical between light and dark modes. Only the **semantic token mappings** change:

| Semantic Token | Light Mode Maps To | Dark Mode Maps To |
|---|---|---|
| `color.bg.default` | neutral-50 | neutral-900 |
| `color.bg.surface` | neutral-100 | neutral-800 |
| `color.bg.elevated` | white | neutral-700 |
| `color.text.primary` | neutral-900 | neutral-50 |
| `color.text.secondary` | neutral-600 | neutral-300 |
| `color.border.default` | neutral-200 | neutral-700 |
| `color.action` | primary-500 | primary-400 |
| `color.action.hover` | primary-600 | primary-300 |

### Surface Hierarchy in Dark Mode

Dark mode reverses the surface hierarchy:

| Layer | Light Mode | Dark Mode |
|---|---|---|
| Base (deepest) | Lightest (white/neutral-50) | Darkest (neutral-950/neutral-900) |
| Surface | Slightly darker (neutral-100) | Slightly lighter (neutral-800) |
| Elevated | Darker still (neutral-200) | Lighter still (neutral-700) |
| Overlay | Dark with opacity | Lighter with opacity |

This creates depth through lightness variation: in dark mode, elements "closer" to the user appear lighter (as if lit from above).

### Contrast Verification

All foreground/background pairings in dark mode must meet WCAG 2.2 AA:

| Content Type | Minimum Contrast Ratio |
|---|---|
| Normal text (< 18pt / < 14pt bold) | 4.5:1 |
| Large text (>= 18pt / >= 14pt bold) | 3.0:1 |
| UI components and graphical objects | 3.0:1 |

**Note:** Contrast rules are applied here as a generation constraint (ensuring generated palettes meet minimum ratios). The full accessibility audit methodology and WCAG success criteria live in HIG's `wcag-baseline.md`.

### Dark Mode Color Adjustments

For primary/semantic colors in dark mode:
- **Shift action colors lighter** (use 400 instead of 500) to maintain visibility against dark backgrounds
- **Reduce chroma slightly** for large colored surfaces to prevent eye fatigue
- **Increase chroma slightly** for small accents to maintain vibrancy against dark surroundings
- **Test colored text** carefully -- colored text on dark backgrounds often needs higher lightness than expected

```json
{
  "color": {
    "$type": "color",
    "primitive": {
      "blue": {
        "50":  { "$value": "oklch(0.97 0.014 250)", "$description": "Lightest blue tint" },
        "100": { "$value": "oklch(0.93 0.032 250)" },
        "200": { "$value": "oklch(0.87 0.058 250)" },
        "300": { "$value": "oklch(0.77 0.099 250)" },
        "400": { "$value": "oklch(0.67 0.144 250)" },
        "500": { "$value": "oklch(0.55 0.180 250)", "$description": "Primary blue" },
        "600": { "$value": "oklch(0.47 0.162 250)" },
```

## Curated Examples

### Linear

**Style:** Minimal cool palette, high-contrast dark mode as default.
- Primary: violet-blue (H ~270)
- Neutrals: cool gray, very low chroma
- Dark mode first -- light mode secondary
- Accent colors: subtle, used sparingly for status indicators
- **Notable:** Surface hierarchy uses very subtle lightness differences (1-2% between layers)

**Source:** https://linear.app/

### Stripe

**Style:** Warm professional palette, subtle gradient accents.
- Primary: indigo-blue (H ~260)
- Neutrals: slightly warm gray (H ~240, very low C)
- Signature gradients: multi-hue blends for marketing, flat colors for product UI
- **Notable:** Different palettes for marketing site (vibrant) vs dashboard (muted)

**Source:** https://stripe.com/

### Figma

**Style:** Playful accent colors, neutral-first base.
- Primary: purple (H ~290)
- Neutrals: true neutral gray (C ~0)
- UI is almost entirely neutral with color used for selection, status, and branding
- **Notable:** Very restrained use of brand purple in product UI -- most color comes from user content

**Source:** https://figma.com/

### Tailwind CSS Default Palette

**Style:** Comprehensive reference scale -- 22 hues, 11 steps each.
- Industry benchmark for palette completeness
- Each hue scale independently tuned (not algorithmically uniform)
- Widely used as a starting point for custom palettes
- **Notable:** Tailwind v3.4+ uses OKLCH internally for palette generation

**Source:** https://tailwindcss.com/docs/customizing-colors
